# 答辩老师视角 200 问与参考答案

## 使用说明

这份文档基于以下三类材料整理：

1. 论文正文：`chap01` 到 `chap06`
2. 当前系统真实实现：Ark 工程中的上传、解析、检索、会话、隐私治理与测试代码
3. 当前答辩 PPT：`/home/lilu/桌面/答辩ppt.pptx`

整理原则不是“泛泛问 200 个 RAG 常识题”，而是站在答辩老师角度，围绕你的论文主线、PPT 展示内容和系统真实实现，去问最有可能出现的问题。

建议使用方法：

1. 先把每个答案的第一句背熟，那是“短答版”。
2. 如果老师继续追问，再补后面两三句。
3. 遇到系统边界问题，不要硬撑，按文中给出的“承认边界 + 说明原因 + 给出后续方向”来答。

## 一、选题背景与研究意义（1-15）

1. 问：你为什么选择“基于 RAG 的企业私有文档智能问答系统”这个题目？

答：因为企业私有文档问答不是单纯的“让模型回答问题”，而是同时受到文档持续更新、权限边界、多轮追问和内容安全的约束。纯大模型参数记忆在这类场景里容易出现幻觉、无依据回答和越权风险，所以需要用工程化 RAG 来解决。这个题目既有明确的应用价值，也有系统设计层面的研究空间。

2. 问：为什么你聚焦的是“企业私有文档”，而不是通用开放域问答？

答：因为开放域问答主要关心答得准不准，而企业私有文档场景还必须关心“能不能访问”“能不能外发”“多轮追问会不会串文档”。换句话说，企业场景的核心不是知识量大，而是约束多、责任重、错误代价高。正因为约束更复杂，系统设计问题才更突出。

3. 问：你的论文想解决的核心问题到底是什么？

答：核心问题是如何在企业私有文档场景下，同时兼顾问答有效性、授权一致性和内容安全。进一步拆开看，就是四个问题：可靠接入、权限前置、多轮会话控制和隐私治理。我的系统也是围绕这四个问题展开设计和实现的。

4. 问：为什么不能直接依赖大模型参数知识来回答？

答：因为企业内部文档具有时效性、专属性和权限边界，这些知识通常不在模型预训练参数里，即使模型“知道一点”，也无法保证是最新版本。更关键的是，纯参数回答通常不可追溯，老师一问“依据是什么”，就很难解释。RAG 的意义就在于把回答建立在外部证据之上。

5. 问：你论文里说的“可信”具体指什么？

答：这里的“可信”不是一个抽象词，而是三个层面的综合要求。第一是回答要尽量基于证据而不是猜测；第二是检索边界必须和访问控制边界一致；第三是敏感内容不能在索引、外发、历史或日志中失控传播。也就是说，可信包含正确性、授权一致性和安全性。

6. 问：为什么你的研究不只谈“回答效果”，还强调“授权一致”？

答：因为在企业场景里，越权比答错更严重。答错可能影响体验，越权可能直接构成安全事故。很多系统是在检索后再裁剪结果，但这样未授权内容已经进入候选集甚至重排阶段了，所以我把权限过滤前置到召回阶段。

7. 问：为什么“内容安全”会成为这篇论文的主线之一？

答：因为企业文档经常包含身份证号、手机号、邮箱、银行卡、凭据和内部敏感信息。只做问答效果、不管内容传播路径，在实际场景里是无法落地的。所以我把隐私治理从一个附属功能提升为贯穿接入、索引、Prompt、历史、输出和日志的全链路机制。

8. 问：文档持续更新为什么会构成一个研究问题？

答：因为企业知识库不是静态语料，而是不断新增和变更的。只要上传、解析、切块、向量化、索引写入这条链路不稳定，知识库就会长期处于“文件到了但查不到”的状态。这样再好的检索算法也没有意义，所以文档接入可靠性必须先解决。

9. 问：你认为从单轮问答到多轮问答，难点增加在哪里？

答：难点不在“能不能记住上文”，而在“该不该继承、继承什么、继承到什么范围”。多轮追问里经常有省略主语、代词指代和长程回溯，如果简单把历史全拼进去，很容易造成语义漂移。我的工作重点就是把这个承接边界显式管理起来。

10. 问：RAG 最初是为缓解幻觉提出的，你为什么还要强调工程系统化？

答：因为“缓解幻觉”只是 RAG 的起点，不是终点。真正落地到企业里，系统还必须处理上传、解析、权限、会话、隐私和运维问题。也就是说，RAG 在工程场景里已经不只是 Prompt 技巧，而是完整知识系统。

11. 问：学术界的 RAG 基准和你论文里的场景，有什么本质差别？

答：很多学术基准默认语料已经静态准备好、权限边界也不存在，重点放在检索和生成本身。而我面对的是动态文档接入、多组织访问控制和多轮安全问答，所以系统边界更完整。换句话说，学术 RAG 更像“已建好库之后怎么问”，而企业 RAG 还要先解决“库怎么可靠地建起来”。

12. 问：为什么企业私有文档场景比公开搜索复杂？

答：因为公开搜索主要处理相关性，企业私有文档同时处理相关性、权限和敏感性。两个用户问同样的问题，结果也可能不同，因为他们可见的文档范围不同。再加上多轮追问和来源追溯，系统复杂度会明显提升。

13. 问：你的论文更偏算法还是更偏系统？

答：更准确地说，它是一篇系统设计与实现型论文，而不是单点算法创新论文。它的贡献不在提出一个全新的理论模型，而在于把可靠接入、权限前置、会话编排和隐私治理整合成一个可运行、可验证的工程系统。这个定位我在答辩时会明确说清楚。

14. 问：这个题目对企业知识管理有什么现实意义？

答：它可以把分散在 PDF、Word、Excel、TXT 里的文档知识转成可检索、可追溯、可授权的知识服务。企业人员不需要手工翻文档，而是可以在权限范围内直接提问并获得来源支持的回答。对知识复用、检索效率和合规管理都有现实价值。

15. 问：如果一句话概括你的研究价值，你会怎么说？

答：我的价值不是单纯把大模型接上企业文档，而是把“能回答”做成“能可靠接入、能按权限回答、能多轮稳定承接、能控制敏感信息传播”的完整系统。

## 二、国内外研究与论文定位（16-30）

16. 问：当前 RAG 研究已经发展到什么阶段？

答：早期主要解决“先检索再生成”的基本范式，后来逐渐发展到查询改写、主动检索、自反思检索和混合检索。近两年更明显的趋势是从算法模块走向框架化和工程化。也就是说，研究重点正在从“单次检索效果”延伸到“系统如何稳定运行”。

17. 问：你认为主流 RAG 论文最大的共性局限是什么？

答：它们大多默认知识源已经完成清洗、切分和索引，评测对象是静态语料库。这个前提在学术 benchmark 里成立，但在企业私有文档场景里通常不成立。文档接入可靠性、权限边界和历史承接这些问题，在很多论文里并不是主问题。

18. 问：为什么你在论文里同时提 DPR 和 BM25？

答：因为它们分别代表稠密检索和词法检索两条主线。DPR 说明语义表示在问答中的价值，BM25 则说明传统词法检索在术语、编号和精确匹配场景仍然很强。我的混合检索设计正是建立在这两类能力互补的认识上。

19. 问：你为什么还会提到 HyDE、FLARE、Self-RAG 这些工作？

答：因为它们代表了 RAG 研究从静态检索向动态组织演进的不同方向。HyDE 关注查询改写，FLARE 关注生成过程中按需检索，Self-RAG 关注模型自我反思和检索决策。引用这些工作，是为了说明我并不是孤立地做系统，而是在已有研究脉络上选择适合企业场景的工程实现路径。

20. 问：RAGLab、FlexRAG、RAGVA 这些框架对你有什么启发？

答：它们给我的核心启发不是某个具体算法，而是模块化和可替换性。也就是把检索器、模型、Prompt 规则和运行配置解耦，不让系统被单一组件绑死。我的 Provider 抽象和配置化设计就是在这个思路上落地的。

21. 问：现有企业知识问答系统为什么还不够？

答：很多现有系统能做“导入一批文档后问答”，但对持续接入、断点续传、权限前置过滤和多轮会话控制支持不足。也就是说，它们更像一个可演示的问答原型，而不是可持续运行的企业知识服务。我的论文重点就在于补这些工程短板。

22. 问：现有研究在文档接入方面的不足是什么？

答：多数研究把文档接入当成一次性离线预处理，不讨论中断恢复、状态一致性和后台异步处理。对企业来说，这意味着大文件上传失败或长任务阻塞时，知识库很难稳定维护。我在系统里把上传确认、异步入库和状态恢复分开处理，就是针对这个不足。

23. 问：现有研究在权限控制方面的不足是什么？

答：很多系统把权限控制放在检索之后，或者只做接口层面的文件可见性判断。这样做的问题是未授权文档可能已经进入候选集，甚至影响重排和上下文拼装。我的系统把权限条件直接注入 Elasticsearch 检索过滤中，在召回阶段就限制边界。

24. 问：现有研究在多轮会话方面的不足是什么？

答：不少系统直接拼接最近若干轮历史，默认历史越多越好。但企业文档问答里，历史过多反而会带来话题漂移和错误继承。我的系统用轮次分类、工作记忆和选择性历史注入来解决这个问题。

25. 问：现有研究在隐私治理方面的不足是什么？

答：很多工作把安全理解为权限控制，较少细分内容本身在索引、外发、历史和日志中的传播风险。也就是说，能否访问一份文档，不等于文档里的每一类内容都能直接外发。我的论文把内容治理单独作为一条系统链路设计出来。

26. 问：你的工作和“把现成框架搭起来”有什么区别？

答：区别在于我不是简单组合几个现成组件，而是围绕企业私有文档场景重新定义了系统边界和关键控制点。比如断点续传和三层一致性、检索前置权限过滤、轮次驱动的会话控制、分阶段隐私治理，这些都不是开箱即用框架自动给出的。它更像是有明确问题定义的工程研究，而不是组件堆叠。

27. 问：你的论文算不算提出了新算法？

答：严格说，不是提出全新的基础算法，而是提出了面向企业私有文档场景的工程机制和系统组合方案。我的贡献在于把多个本来分散的问题组合成一套可解释、可实现、可测试的架构。答辩时我会把它定义为“系统设计创新和工程机制创新”。

28. 问：你的权限设计和传统 RBAC 有什么关系？

答：我是在 RBAC 基础上扩展了组织标签维度。RBAC 解决的是角色差异，但企业内部同一角色也可能属于不同组织、看不同文档，所以仅靠角色不够。我的设计把 role、owner、isPublic 和 orgTag 共同纳入权限计算。

29. 问：为什么还要加组织标签，而不是只用角色和用户归属？

答：因为企业组织往往是层级结构，同一角色在不同部门的可见范围并不一样。组织标签能够把组织边界和文档边界对应起来，还能表达父子组织继承关系。这样权限模型更符合真实企业场景。

30. 问：如果让你总结这篇论文的学术定位，你会怎么说？

答：它的定位是面向企业私有文档问答场景的系统设计与实现研究。论文的重点不是追求单项 SOTA，而是围绕真实部署问题提出结构化解决方案，并用系统测试和对照实验验证这些设计是否有效。

## 三、需求分析与应用场景（31-45）

31. 问：为什么你选择企业公告及其附件作为测试语料？

答：因为这类语料具有企业私有文档场景的典型特征：篇幅长、格式混杂、数字和专业术语多、问答 often 需要来源依据。它既适合作为问答语料，也能体现权限、检索和表格处理问题。对论文的验证比较有代表性。

32. 问：为什么测试语料要同时包含 PDF、Word、Excel 和 TXT？

答：因为真实企业知识库不会只有一种格式。PDF 代表正文报告，Word 和 Excel 常见于公开附件和表格类资料，TXT 则代表轻量文本。只有覆盖多格式，才能证明系统不是只对单一文档类型有效。

33. 问：你在第二章里最核心的功能需求有哪些？

答：可以概括为四组：用户与组织管理、文档接入与知识入库、检索与问答、会话交互与文档服务。前两组解决“知识怎么进来”，后两组解决“知识怎么安全地被用起来”。这四组需求构成了系统的主干。

34. 问：最核心的非功能需求有哪些？

答：主要是性能效率、安全性、可靠性和可维护性。因为企业系统不能只在 demo 场景能跑，还要看时延、越权风险、恢复能力以及后续维护成本。第五章就是按这些非功能要求去做验证的。

35. 问：为什么用户和组织管理会被放进这篇论文？

答：因为权限控制的前提就是先有清晰的主体信息。用户角色、主组织标签、有效组织标签集合，都会直接影响后续检索边界和文件服务边界。没有这部分，后面的权限一致性就无从谈起。

36. 问：为什么文档服务是系统需求的一部分，而不只是问答？

答：因为企业使用知识系统时，不会只停留在“问一句答一句”。他们还需要查看文件列表、预览、下载、删除和历史追溯，这些都和权限和一致性直接相关。问答只是入口，文档服务是完整知识闭环的一部分。

37. 问：为什么来源追溯必须作为需求单独提出？

答：因为企业问答里老师或用户经常会反问“这个结论是依据哪份文件、哪一段”。如果系统不能回溯来源，即使答案看起来正确，也很难被信任。我的系统通过来源编号和文件映射来解决这一点。

38. 问：为什么 History API 和历史会话回读也算正式需求？

答：因为多轮问答如果没有稳定的历史读取和会话回放，就没法验证会话管理的有效性。History API 不只是给前端显示历史，它也是检验会话状态是否真实持久化的一部分。它连接的是交互体验和系统一致性。

39. 问：为什么你把 P95≤1500 ms 作为在线时延目标？

答：因为平均时延容易掩盖长尾，而用户在实际使用中更容易感受到长尾卡顿。P95 既能体现系统的稳定性，又比极端值更适合作为工程目标。论文里这个阈值是为了兼顾企业单机部署的现实约束和可用体验。

40. 问：为什么可靠性要作为独立需求，而不是并入性能？

答：因为性能说的是“快不快”，可靠性说的是“断了以后能不能恢复、状态会不会错乱”。一个系统可能很快，但上传中断就报废，这在企业场景里仍然不可接受。所以我单独验证了续传、解析、索引和入库一致性。

41. 问：为什么可维护性也要验证？

答：因为企业系统不是一次性交付，而是要持续演进。模型可能换、Prompt 规则可能调、检索参数可能改，所以如果模块边界混乱，后续维护成本会很高。我的 Provider 抽象、配置解耦和结构符合性检查，就是在回应这个需求。

42. 问：为什么上传固定分片大小是 5 MB？

答：这是在上传稳定性、网络重传成本和服务端处理开销之间做的工程平衡。分片太大，重传成本高；分片太小，状态管理和对象数量都会显著增加。5 MB 是比较稳妥的工程配置。

43. 问：为什么 1 MB 以下的小文件走直通处理？

答：因为小文件如果也全部进 Kafka，会引入额外排队和调度开销，收益不高。直通处理能减少这类轻量任务的额外等待，而大文件再走异步链路。这个分流的本质是让不同量级的任务走不同成本路径。

44. 问：为什么问答上下文默认只取 Top-5？

答：因为上下文过多会带来两类问题，一类是模型输入长度和成本上升，另一类是噪声证据变多。Top-5 既能覆盖多个相关块，又能控制上下文规模。当前正式实验也就是在这个配置下完成的。

45. 问：你的系统边界里没有做什么？

答：当前系统没有做到完整的多模态理解，也没有做单元格级 Excel 语义建模和生产级大规模分布式部署。我的论文更聚焦文本类企业文档的工程化 RAG 验证。这个边界我会在局限性里主动说明。

## 四、总体架构与技术选型（46-70）

46. 问：你的系统总体架构是什么样的？

答：可以概括为前后端分离、多存储协同、模型访问解耦的分层架构。前端负责上传和交互，后端负责编排，MySQL、Redis、MinIO、Elasticsearch 和 Kafka 分别承担不同数据职责。整体上围绕文档接入链路和问答链路两条主线组织。

47. 问：为什么采用前后端分离架构？

答：因为上传、问答、历史管理和权限服务都需要清晰的接口边界。前后端分离有利于独立开发、独立迭代，也方便前端用 WebSocket 做流式交互。对答辩而言，这也能体现系统模块化设计而不是脚本式原型。

48. 问：为什么前端选 Vue 3 + Vite？

答：主要是因为它适合快速构建交互界面，同时支持组件化和较好的开发体验。答辩里这不是重点，但可以说它满足了文件上传、知识库管理和流式问答界面的实现需要。我的论文重点不在前端框架创新，所以这里强调“适配需求、开发效率高”就够了。

49. 问：为什么后端用 Spring Boot + WebSocket？

答：Spring Boot 适合组织 REST、权限过滤、服务分层和中间件集成；WebSocket 适合流式回答场景，因为它比轮询更自然地支持增量输出和完成通知。对于我的系统来说，这个组合能兼顾普通接口和实时问答链路。

50. 问：为什么用 MySQL？

答：因为文件记录、分片记录、用户组织信息和解析后的结构化分块，都需要稳定持久化和较好的结构化查询能力。MySQL 在这类事务性和关系型数据上更合适。它在系统里承担的是“元数据真相源”的角色。

51. 问：为什么用 Redis？

答：因为上传位图、当前会话 ID、原始历史和工作记忆都属于高频、短期、低延迟读写的数据。Redis 在这种状态型数据上比关系库更合适。尤其上传分片状态和会话状态，要求毫秒级访问。

52. 问：为什么用 MinIO？

答：因为原始文件和分片对象本质上都是对象数据，不适合塞进关系库。MinIO 既适合保存原件，又支持对象级操作和预签名访问，非常适合作为原文件对象存储层。它让系统保留了“原始事实源”。

53. 问：为什么用 Elasticsearch？

答：因为系统既需要 BM25 词法检索，也需要向量检索和过滤条件，同时还要支持文件名、标题和归一化字段。Elasticsearch 比单一向量库更适合我这种混合检索场景。它在系统里承担的是“检索索引层”。

54. 问：为什么用 Kafka？

答：因为上传确认和后台入库的耗时量级不一样。Kafka 可以把大文件解析、切块、向量化这些重任务从在线请求里拆出来，同时支持重试和死信。它本质上是把长任务变成可恢复、可观测的异步流水线。

55. 问：为什么要做聊天模型和向量模型的 Provider 抽象？

答：因为生成调用和向量化调用在延迟、吞吐和失败模式上都不一样，未来也可能换模型供应商。通过 Provider 抽象，上层业务只依赖统一接口，不直接耦合某家模型服务。这样模型切换时影响范围更小。

56. 问：为什么不把所有数据都放进一个数据库里？

答：因为不同数据的访问模式完全不同。元数据需要关系型结构，上传位图和会话状态需要高速缓存，原文件需要对象存储，检索又需要专门的搜索索引。多存储协同不是复杂化，而是按数据职责拆分。

57. 问：你的系统两条主链路分别是什么？

答：第一条是文档接入链路，从上传、合并、解析、切块、向量化到索引写入；第二条是问答链路，从提问、轮次判断、权限检索、Prompt 组装到流式回答。前者解决“知识怎么进入系统”，后者解决“知识怎么被安全地使用”。

58. 问：为什么文档接入必须异步化？

答：因为解析、切块和向量化是重任务，尤其大文档可能要花数秒到十几秒。如果都阻塞在用户请求里，上传体验会很差，而且失败恢复也困难。异步化的意义是解耦在线确认和后台知识化处理。

59. 问：为什么权限过滤要前置到检索阶段？

答：因为如果先全库召回再裁剪，未授权文档已经进入候选集，理论上就可能影响重排甚至上下文拼装。前置过滤让未授权文档从一开始就不参与召回。这样检索边界和授权边界才真正一致。

60. 问：为什么要做混合检索，而不是只选一种？

答：因为企业文档既有“术语、编号、金额”这种需要精确匹配的内容，也有“语义改写、同义表达”这种需要语义召回的内容。单一检索路径很难同时兼顾两类场景。混合检索就是为了把两类优势叠加起来。

61. 问：为什么 Prompt 要做成结构化分层？

答：因为规则、历史、工作记忆和检索证据本质上不是同一种信息。把它们混成一段文本，模型容易混淆“哪部分是约束、哪部分是事实、哪部分只是上下文”。分层装配能让这些角色更清晰。

62. 问：为什么会话状态控制要独立成一个模块？

答：因为多轮问答最怕的不是“模型忘了”，而是“模型乱继承”。如果不先判断轮次类型，再决定历史是否参与和检索范围是否收缩，系统很容易在追问中串题。这个模块就是在模型调用前做边界管理。

63. 问：为什么隐私治理必须单独成链，而不是某个接口顺手处理一下？

答：因为敏感信息的传播路径不止一个出口，它会经过入库、embedding、Prompt、历史、输出和日志多个环节。只在最后一层做过滤，很可能前面已经泄露了。单独成链，才能把治理前移到整个过程。

64. 问：为什么对象存储和检索索引必须分开？

答：因为原文件和检索块是两种完全不同的视图。对象存储保留原件，索引保存的是切块后的检索视图。只有分开保存，系统才能同时兼顾可追溯和可检索。

65. 问：为什么关系型数据和索引数据也要分开？

答：因为关系型数据强调事务和结构化约束，而检索索引强调召回、排序和过滤。把它们混在一起会让两边都不合适。分开以后，关系库做元数据，ES 做搜索，各司其职。

66. 问：为什么你的系统里有这么多缓存？

答：因为查询向量、检索结果、文件名、用户信息和组织标签都存在明显热点，如果每次都重新算或重新查，时延会更高。缓存的目标不是改变业务语义，而是减少重复计算和重复访问。尤其在并发场景下，缓存能直接提升吞吐。

67. 问：为什么还要做 in-flight 去重？

答：因为缓存只对“已经完成”的结果有效，而并发场景下同一请求可能同时在路上。in-flight 去重可以把这些并发中的重复请求合并掉，避免多个线程同时打向量模型或检索服务。它对高并发热点问题特别有价值。

68. 问：为什么问答链路选 WebSocket，不用 HTTP 轮询？

答：因为模型回答是增量产生的，WebSocket 更适合持续推送 chunk 和完成通知。轮询不仅效率低，还会让前端状态更复杂。对于流式问答来说，WebSocket 是更自然的实现方式。

69. 问：你为什么采用单机真实环境测试，这样是否太弱？

答：单机环境并不是为了追求规模，而是为了贴近中小型企业私有部署的现实边界。我的论文目标是验证系统架构在受控条件下是否成立，而不是一开始就宣称大规模生产级能力。这个定位在结论和局限性中都说得很清楚。

70. 问：如果以后走向更大规模部署，架构上要改什么？

答：最先要增强的是可观测性、故障恢复和分布式部署能力。包括 Kafka、ES、对象存储和应用服务的多节点扩展，缓存一致性管理，以及更完整的监控和告警。当前架构为这些扩展留了方向，但论文还没有把它们做到生产级。

## 五、文档接入与知识入库（71-95）

71. 问：你的上传链路完整流程是什么？

答：前端先按 5 MB 分片上传，后端验证文件记录和分片状态，把分片写入 MinIO，同时在 Redis 和数据库记录状态。分片齐全后执行合并，生成 `merged/fileMd5` 原文件对象，然后按文件大小选择直通处理或投递 Kafka，最后进入解析、切块、向量化和索引写入。

72. 问：为什么要做断点续传，而不是失败后重传整文件？

答：因为企业文档里大文件很多，整文件重传成本高、失败概率也更大。断点续传可以把失败影响限制在少量缺失分片上。这样既提高用户体验，也提高链路可靠性。

73. 问：为什么文件要用 MD5 做标识？

答：因为 MD5 能提供稳定的内容级标识，适合用于上传去重、分片归属和合并对象命名。尤其在对象存储里，用内容指纹比用文件名更稳，因为文件名可能重复。我的系统里很多状态关联都围绕 `fileMd5` 展开。

74. 问：为什么分片状态要用 Redis bitmap 表示？

答：因为它天然适合表示“第几个分片是否已上传”这种布尔状态，而且读写都很快、内存开销也低。相比一条分片一条记录去查，bitmap 更适合高频状态探测。它让前端快速知道哪些分片需要补传。

75. 问：既然有 Redis bitmap，为什么还要数据库里的 ChunkInfo？

答：因为 Redis 适合短期高频状态，不适合承担长期结构化真相。ChunkInfo 记录了分片索引、分片 MD5 和存储路径，是后续合并和一致性核验的重要依据。两者的角色不同，不能互相替代。

76. 问：为什么还要去 MinIO 检查对象是否真的存在？

答：因为只看 Redis 或数据库都可能出现“状态写了，但对象其实没有”的不一致情况。MinIO 对象存在性检查是为了确认物理对象真的落盘了。也就是说，系统不会盲信任何单一状态源。

77. 问：为什么分片对象路径要设计成 `chunks/{fileMd5}/{chunkIndex}`？

答：因为这个路径天然表达了“这是谁的第几个分片”，便于定位和合并。以 MD5 分组还能避免不同文件同名冲突。路径设计本身就是一种简单但稳定的状态组织方式。

78. 问：为什么合并对象路径用 `merged/{fileMd5}`，而不是用原文件名？

答：因为文件名可能重复，但 MD5 更稳定、唯一。用文件名作为对象名会有同名覆盖风险，而用 MD5 可以把不同内容的同名文件区分开。对后续回源和一致性校验也更方便。

79. 问：为什么不是最后一个分片到了就立刻合并，而是还要先校验？

答：因为“最后一个分片到了”不等于“所有分片都可靠存在”。系统还要确认记录数量一致、对象都真实存在、状态没有丢失。先校验再合并，能把错误阻断在合并前。

80. 问：为什么合并前要逐个 `statObject` 检查分片？

答：因为数据库里有记录不代表对象一定还在，尤其在异常情况下可能出现逻辑记录和物理对象不一致。逐个检查虽然会增加一点线性开销，但能显著提高合并成功的可靠性。当前论文也诚实承认这一步未来还可以并行化优化。

81. 问：为什么合并成功后要删除分片对象？

答：因为分片对象只是传输过程中的中间态，长期保留会浪费对象存储空间，也增加状态混乱风险。合并成功后系统已经有稳定的原文件对象，所以分片可以清理。这里的删除是“合并后清理中间产物”。

82. 问：为什么合并后返回预签名 URL？

答：因为这样可以在不暴露存储系统内部细节的前提下，支持临时访问原文件对象。它适合预览、下载和人工复核场景。既保留访问能力，又控制了访问时效。

83. 问：为什么小文件走同步直通，大文件走 Kafka 异步？

答：因为小文件的解析和入库成本低，如果也进入异步队列，调度开销反而占了大头。大文件则相反，重处理链路长，必须异步化才不会阻塞前台请求。这个分流是基于任务成本差异做的工程优化。

84. 问：为什么要先把状态写成“已合并”，再投递 Kafka？

答：因为系统采用的是“状态先行”的一致性策略。这样即使后续消息投递失败，也能知道文件已经到达哪个阶段，再由重试或补偿机制继续推进。它比“先投消息再记状态”更容易恢复。

85. 问：为什么要配置 Kafka 重试和死信队列？

答：因为解析和向量化涉及外部模型、对象存储和多格式文档，失败是现实存在的。重试可以解决瞬时故障，死信队列则保证长期失败的任务不会悄悄丢失。这样系统失败后仍然可追踪、可补偿。

86. 问：如果 Kafka 消息发送失败怎么办？

答：系统会依赖消息发送重试以及后续的补偿处理，避免任务直接消失。更重要的是，文件状态已经先推进到“已合并”，因此这不是不可恢复的黑洞状态。答辩时可以强调：系统优先保证状态可追踪。

87. 问：消费者是如何下载文件的？

答：消费者优先按 `merged/fileMd5` 从 MinIO 下载原文件对象。如果这个路径不可用，再回退到任务中记录的文件路径或 URL。也就是说，MinIO 原文件是首选回源方式。

88. 问：为什么优先从 MinIO 对象回源，而不是直接用文件路径？

答：因为 MinIO 里的 `merged/fileMd5` 是系统内部统一、稳定的原件表示，而 filePath 可能是历史路径、外部 URL 或兼容字段。优先对象回源能提高一致性，也更符合当前系统对原件的管理方式。

89. 问：为什么合并后还要保留原文件，而不是只保留切块和索引？

答：因为索引只是检索视图，不是原始事实源。保留原文件可以支持重新解析、重新向量化、人工复核、预览下载和故障回源。没有原文件，系统就会退化成“只有索引、没有原件”的状态。

90. 问：如果 MinIO 上传成功了，但 Redis 标记失败，会怎样？

答：系统不会因为 Redis 标记失败就把已上传分片回滚，因为真正更核心的事实是对象已经存在于 MinIO。后续仍可以通过数据库记录和对象存在性检查补偿这个状态差异。也就是说，Redis 不是唯一真相源。

91. 问：如果 Redis 说已经上传了，但数据库里没有对应分片记录怎么办？

答：系统不会直接跳过，而是会进一步检查 MinIO 中该分片对象是否真实存在。若对象存在，就补写数据库记录；若对象不存在，就按未上传重新走上传流程。这体现了系统的三层一致性思路。

92. 问：如果合并时发现分片数量不匹配怎么办？

答：系统会直接拒绝合并，不会带着缺片进入后续入库流程。前端需要重新拉取上传状态，只补传缺失分片后再重试。这样做是为了把错误消灭在合并前。

93. 问：如果用户上传的是不支持的文件类型怎么办？

答：系统会在上传阶段由 `FileTypeValidationService` 拦截，不让它进入解析和向量化链路。因为有些格式本身无法稳定解析为知识文本，强行进入后续流程没有意义。这个设计也是在保护系统稳定性。

94. 问：如果扩展名和 MIME 类型明显不一致怎么办？

答：系统会视为风险输入并拒绝上传，避免伪装文件进入处理链路。对于 `application/octet-stream` 这类通用 MIME，系统会兼容放行；但对明显冲突的情况会阻断。这是在兼顾兼容性和安全性。

95. 问：文档删除之后，系统会清理哪些内容？

答：会同步清理数据库记录、对象存储对象、向量记录和 Elasticsearch 索引，避免出现“文件删了但还能被召回”的情况。论文里的 D4 功能验证就是专门验证这个一致性。它体现的是生命周期闭环管理。

## 六、解析、切块、向量化与检索实现（96-120）

96. 问：文档解析的完整流程是什么？

答：消费者拿到原文件后，先判断是否走快速文本通道或 Tika 通道，然后流式提取文本。接着系统按父块缓冲、子块语义切分的方式生成知识分块，再做脱敏、归一化和入库。之后向量化服务再把这些分块写入 Elasticsearch 索引。

97. 问：为什么要区分快速文本路径和 Tika 路径？

答：因为纯文本类文件没必要都走复杂的格式识别流程。对 `md/txt/csv/json/xml/html/log` 等文件直接流式读取，开销更小、效率更高；复杂格式再交给 Tika。这样能减少不必要的解析成本。

98. 问：为什么解析要采用流式方式，而不是整文件一次性读入？

答：因为企业文档里有很多大 PDF 和长报告，一次性读入容易造成内存压力甚至 OOM。流式处理可以把内存占用控制在更可预测的范围内。它是为了让系统在真实文件规模下仍然稳定运行。

99. 问：什么叫“父块缓冲、子块切分”？

答：父块是流式解析阶段按较大缓冲阈值先积累出来的一段文本，子块则是父块内部进一步切成适合索引和 embedding 的语义块。也就是说，父块是解析过程中的工程缓冲单位，子块才是检索阶段真正使用的知识单元。这个两级设计兼顾了流式处理和语义切分。

100. 问：为什么解析前还要检查内存阈值？

答：因为大文件解析是最容易把内存打满的环节之一。系统先检查内存占用，如果超过阈值会尝试回收，仍不够再拒绝继续处理。这个机制是为了保护系统稳定性，而不是一味硬跑。

101. 问：语义切块具体是怎么做的？

答：系统先按段落切分，再识别是否为表格型段落；普通长段再按句子切分，超长句再用 HanLP 分词寻找合理断点。最后把这些语义单元组装成适合向量化和检索的块。也就是说，它不是简单按固定字符数硬切。

102. 问：为什么最终分块长度设置为 512 字符？

答：这个值是在“语义完整性”和“向量化预算”之间做的平衡。企业公告类文档的大多数自然段在这个长度范围内还能保持较好完整性，同时不会把单次 embedding 输入拉得过长。它是结合语料特征和工程成本确定的。

103. 问：为什么当前系统没有做固定 overlap？

答：因为当前设计更强调按段落、句子和表格行等显式语义边界切分，先尽量减少机械截断。固定 overlap 虽然能缓解跨块断裂，但也会放大索引规模和重复片段，对重排序稳定性有影响。所以当前版本先采用“语义切块、无固定 overlap”的方案。

104. 问：如果同一主题被切到两个块里，会不会导致语义断裂？

答：有这个可能，所以系统才尽量沿段落、表格和句子边界切，而不是暴力截断。当前缓解手段是语义边界切分、多结果 topK 检索和多轮追问的文档范围收缩。但我也会承认，当前还没有做 chunk overlap 和父子块联动检索，这属于后续优化空间。

105. 问：检索时两个相关块会不会一起被召回？

答：如果两个块都和问题足够相关，并且都排进前面的 topK，它们是可以一起进入上下文的。系统不是只取一个块，而是会返回多个检索结果后再组织授权证据。风险在于其中一个块得分不够高时，可能只看到部分信息。

106. 问：系统如何处理表格型段落？

答：系统会先判断一个段落是否像表格，比如是否包含多行、制表符、分隔符或数值信号。对表格段，不是按普通段落切，而是尽量按行组合保留局部结构。这样做能减少把表格行内关系打散的问题。

107. 问：Excel 在系统里是如何被解析的？

答：底层依赖 Apache Tika/POI 统一抽取 Excel 中的可见文本，再在上层做表格型段落识别和切块。也就是说，它已经支持 Excel 文本问答和表格样段检索，但不是完整的单元格级结构化建模。这个说法要答得准确，不要夸大。

108. 问：有没有专门写 Excel 解析器？

答：当前没有单独写一套 Excel 专属解析器，而是复用 Tika/POI 的统一文档解析能力。这样做降低了实现复杂度，也保证了不同格式的统一处理路径。但代价是 Excel 语义结构保留还不如专门的表格理解系统细。

109. 问：如果文档解析失败，系统会怎么处理？

答：如果失败属于 Tika、POI、SAX 这类解析链路异常，系统会触发降级策略，不直接让整份文件报废。具体做法是生成一个元数据保底分片，用于后续检索和人工复核。这样至少保住“可发现性”，而不是完全丢失。

110. 问：如果解析后没有抽出正文怎么办？

答：系统会写入一个保底分片，至少包含标题、文件名以及“正文解析为空”的说明。这样文档仍能通过标题被检索到，也能进入人工复核流程。它体现的是“尽量保住可检索性”的设计取向。

111. 问：解析后的数据结构是什么？

答：解析阶段保存的是 `DocumentVector` 结构，不只是文本，还包含 `safeTextContent`、`normalizedTextContent`、用户、组织、公开性以及敏感级别等字段。进入检索层后则会转成 `EsDocument`，再加上向量、文件名和标题等索引字段。换句话说，解析结果是“带治理元数据的知识块”。

112. 问：为什么要同时保存 safeText 和 normalizedText？

答：因为两者服务的目标不同。safeText 解决的是隐私和外发控制问题，normalizedText 解决的是检索时数字、单位、符号和噪声表达不稳定的问题。它们一个偏安全视图，一个偏检索视图。

113. 问：为什么要做数字和单位归一化？

答：因为企业文档里金额、比例、“万/亿”等表达非常多，同一个含义可能有多种写法。归一化能让 BM25 和重排在更稳定的文本表示上工作，从而提升数字型查询的匹配效果。它是混合检索适配财务类文档的重要细节。

114. 问：向量化阶段的流程是什么？

答：系统先从数据库取出解析后的分块，优先选择 safeText，然后批量调用 embedding 模型生成向量。之后把文本、向量、文件名、标题、权限字段和敏感元数据一起构造成 ES 文档写入索引。这样检索层拿到的是完整检索单元。

115. 问：为什么向量化优先基于 safeText，而不是原始文本？

答：因为 embedding 往往依赖外部模型服务，如果直接把原文送出去，敏感信息风险就会在索引构建阶段提前发生。用 safeText 可以在保留主要语义的同时控制外发风险。这个设计体现了隐私治理前移。

116. 问：为什么论文里向量维度是 2048？

答：因为当前实验环境使用的是 `text-embedding-v4`，在本论文配置下输出 2048 维向量。这个维度不是我拍脑袋定的，而是由所选模型决定的。系统本身通过抽象接口屏蔽了底层维度细节。

117. 问：为什么 embedding 还要做缓存？

答：因为重复文本在企业文档和高频查询里并不少见，如果每次都重新调外部模型，成本和时延都会上升。缓存可以复用已有结果，减少不必要的外部调用。它直接针对 embedding 成本高这个瓶颈。

118. 问：为什么还要做自适应并发限制器？

答：因为 embedding 是典型的外部依赖，时延和失败率会随负载波动。固定并发在轻载时可能浪费吞吐，在重载时可能把外部服务压垮。自适应限流让系统在健康时逐步提速，在高时延或 429、5xx 时主动降速。

119. 问：为什么说 embedding 是当前系统的主要瓶颈？

答：因为正式入库实验里，embedding 平均耗时 1559.9 ms，占消费者处理总时长的 77.4%，而且 197 个样本的主瓶颈都落在这个阶段。也就是说，解析和 ES 写入都不是当前最重的环节。这个结论有明确实验数据支撑。

120. 问：为什么检索引擎不用专门的向量库，而选 Elasticsearch？

答：因为我的系统不是纯向量召回，而是需要 BM25、向量检索、过滤和重排统一协作。Elasticsearch 在这种混合检索场景里更合适，也便于把权限过滤前置到同一次查询中。它不是“最好”的唯一选择，但和我的系统目标更匹配。

## 七、会话管理、Prompt 与隐私治理（121-145）

121. 问：什么叫轮次路由？

答：就是系统先判断当前输入属于新话题、追问、元问题还是澄清，再决定后续应该怎么处理。它不是给每轮贴标签这么简单，而是决定检索是否开启、历史是否继承、范围是否收缩。它是会话管理的入口控制器。

122. 问：为什么不直接让大模型自己判断是不是新话题？

答：因为如果把这个判断完全交给大模型，系统就很难保证一致性和可解释性。我的系统先用本地规则做轻量判定，把最关键的上下文边界控制在模型之前。这样能减少语义漂移和错误继承。

123. 问：你的轮次类型有哪些？

答：有四类：`NEW_TOPIC`、`FOLLOW_UP`、`META` 和 `CLARIFICATION`。新话题重置上下文，追问承接活动范围，元问题走会话控制分支，澄清则暂停检索先追问用户。每一类都对应不同的后续动作。

124. 问：系统如何判断这是一个新话题？

答：主要看显式新话题标记、是否有明确主语锚点、当前问题是否和已有活动主题脱钩，以及当前会话里是否缺少可承接范围。满足这些条件时，系统就会判为 `NEW_TOPIC`。然后会关闭历史继承、允许全局检索。

125. 问：系统如何判断一个问题是追问？

答：会结合短追问信号、上下文依赖型问题模式、显式历史引用、来源编号引用和当前活动文档范围来判断。如果这些信号说明问题依赖上文，系统会判为 `FOLLOW_UP`，并尝试基于活动文档和关键词改写检索查询。

126. 问：用户说“来源#1 里这个数据为什么这样”，系统是怎么处理的？

答：因为系统在上一轮已经建立了“来源编号 -> 文件 MD5”的映射，所以这里可以直接把 `来源#1` 映射回对应文档。接着系统就会优先做限定文档范围检索，而不是重新全库搜索。这样能让追问更稳、更聚焦。

127. 问：什么是工作记忆？

答：工作记忆不是原始历史，而是系统为了后续承接而提炼出来的结构化上下文。它保存的是当前主题、当前主语、活动文档、关键词和最近答案摘要等。它的目标是“帮助下一轮判断该继承什么”。

128. 问：会话历史和工作记忆有什么区别？

答：会话历史保存的是完整原始消息序列，强调过程记录；工作记忆保存的是结构化摘要，强调后续承接效率。前者更完整，后者更轻量、更可控。把两者分开，是为了防止历史污染当前推理。

129. 问：当前工作记忆里具体存了什么？

答：包括 `activeTopic`、`activeSubject`、`lastStandaloneQuery`、`lastUserMessage`、`lastAnswerSummary`、`lastTurnType`、`updatedAt`、`activeDocumentMd5s`、`activeDocumentNames` 和 `activeKeywords`。这些字段共同表示“当前对话的活动范围”。它不是随便存一些文本，而是有明确结构。

130. 问：为什么选择性历史注入比最近历史拼接更好？

答：因为最近历史拼接只能保留时间上最近的信息，不一定保留语义上最关键的锚点。选择性历史则是在消息预算相同的情况下，优先保留和当前问题最相关的内容，尤其对长程回溯更有效。实验结果也显示它的整体语义通过率最高。

131. 问：如果追问太模糊，但当前没有活动文档范围，系统怎么办？

答：系统不会盲目检索，而是判为 `CLARIFICATION`，先返回澄清提示。这个设计体现的是“宁可先问清楚，也不乱继承”。在企业场景里，这比自信地答错更安全。

132. 问：为什么要把 memory 和 authorized context 分开？

答：因为工作记忆只是系统对话状态摘要，不是证据原文；授权上下文才是当前轮真正允许模型依赖的检索证据。如果把两者混在一起，模型容易把记忆内容误当成事实文档。分开后还可以明确规定冲突时以授权证据为准。

133. 问：turn_summary 有什么作用？

答：它把轮次判断的结果显式告诉模型，比如当前是不是 follow-up、要不要用历史、检索范围是否受限、检索问题是否被改写。这样模型不是在一团文本里“猜上下文”，而是收到结构化的轮次说明。它提升了 Prompt 的可控性。

134. 问：memory_summary 有什么作用？

答：它把工作记忆以摘要形式注入模型，比如当前主题、活动文档、关键词和上一轮答案摘要。这样模型能理解当前对话的活动范围，但又不会接触太多原始历史噪声。它是“轻量承接”的关键。

135. 问：为什么说 authorized context 优先级最高？

答：因为它是本轮经过权限过滤后得到的实际证据，而 history 和 memory 只是辅助信息。如果三者冲突而不设优先级，模型就可能基于旧历史覆盖当前证据。明确“以授权证据为准”是防止事实漂移的关键规则。

136. 问：来源编号映射为什么重要？

答：它一方面支持答案中的可追溯引用，另一方面支持后续用户基于“来源#1”“来源#2”继续追问。这样来源不只是展示层的装饰，而是会话控制的一部分。它把追溯性和多轮性连接起来了。

137. 问：为什么隐私治理要做成分阶段的，而不是统一一个脱敏函数？

答：因为索引、外发、历史、日志和输出面对的风险级别不一样。比如有些内容可以参与内部检索，但不允许送给外部模型；有些内容可以保留安全摘要，但不该出现在日志里。分阶段治理才能精细控制不同环节的暴露面。

138. 问：系统当前识别哪些敏感数据？

答：至少覆盖身份证号、手机号、邮箱、银行卡号、API Key、访问令牌、凭据赋值模式以及企业受限关键词和受限正则模式。对这些内容，系统会生成敏感级别、受限标记和类型摘要。也就是说，不是简单命中一个词就结束，而是会结构化记录治理结果。

139. 问：什么是 `externalModelAllowed`？

答：它表示某段文本在治理后是否允许继续发送到外部模型服务。即使一段内容在内部索引里可以保留安全视图，也不一定允许对外发送。这个字段把“可检索”和“可外发”区分开了。

140. 问：隐私治理具体覆盖哪些阶段？

答：覆盖入库、索引、embedding、Prompt 上下文、会话历史、模型输出、日志和检查接口等多个阶段。也就是说，系统不是只保护最终答案，而是在整条知识链路上都做治理。这样才能真正做到全链路控制。

141. 问：高风险提示是怎么被拦截的？

答：系统会在模型调用前检测用户输入中是否存在“导出原文”“忽略规则”“角色扮演绕过”“编码绕过”等高风险意图。若命中，就直接拒答，不再进入模型。这个入口前置拦截可以显著降低被越狱的风险。

142. 问：为什么不能只在最终输出时再做脱敏？

答：因为那样太晚了。敏感内容可能已经在 embedding、Prompt、历史或者日志阶段泄露出去了。只看最终输出，控制不住中间传播路径。

143. 问：如果用户明确要求“忽略系统提示词”，系统怎么办？

答：系统不会真的忽略规则，而是会把这类请求识别为高风险提示注入尝试，并在模型调用前阻断。即使没有直接阻断，Prompt 中的隐私策略和规则层也会明确要求拒绝此类请求。这个问题可以作为系统安全设计的一个例子来回答。

144. 问：为什么模型输出还要再做一轮复核？

答：因为即使输入受控，模型在生成过程中仍可能重建敏感模式或输出不该展示的内容。输出复核是最后一道保险。它和前面的输入保护不是重复，而是前后呼应。

145. 问：系统如何避免跨会话污染和串答？

答：每个用户会有当前会话 ID，会话历史和工作记忆都按会话隔离存储。再加上引用映射和流式状态也按 session 管理，所以不同会话不会共享上下文。对攻击性提示中试图利用前一会话污染当前回答的情况，也有专门的对抗测试覆盖。

## 八、测试设计与实验结果（146-170）

146. 问：为什么正式实验选了 200 份文档和 400 条查询？

答：因为这个规模已经足以覆盖多种文档类型、文件格式和查询类型，同时又在当前单机部署条件下可控。它比几十份样本更有说服力，又不会夸大到脱离实际环境。论文里所有主结果都统一在这个正式口径下报告。

147. 问：这些查询是怎么设计的？

答：查询不是随便写的，而是覆盖精确匹配、简称缩写、语义改写、属性描述、时间约束、多约束组合等多类场景。这样可以更全面地反映混合检索在真实问答中的表现，而不是只测某一类容易的问题。

148. 问：为什么这篇论文的主评测采用文档级命中率？

答：因为当前测试集里每条查询对应唯一目标文档，所以文档级命中是最直接、最客观的核心指标。它可以避免人工判断多段文本相关性的主观性，同时适合 Recall、MRR 和 nDCG 这类指标计算。这个评测口径和任务定义是一致的。

149. 问：Recall@K 在这里怎么理解？

答：就是目标文档有没有出现在前 K 个结果中。它回答的是“系统有没有把正确文档召回来”。在我的论文里，Recall@5 是主指标之一。

150. 问：MRR@K 的意义是什么？

答：MRR 看的是目标文档排得靠不靠前，不只是有没有召回。排名越靠前，倒数值越高，MRR 就越大。它更能反映“系统有没有把正确文档放到用户最容易看到的位置”。

151. 问：nDCG@K 在你的任务里有什么意义？

答：它对靠前位置的相关结果给予更高权重，强调排序质量。虽然当前任务每条查询只有一个目标文档，但 nDCG 仍能补充体现排序位置的好坏。它和 MRR 一起，用来评估排序敏感性。

152. 问：为什么你还测了 TTFT？

答：因为 TTFT 反映的是用户感知到“系统开始回答”的等待时间，而不仅仅是最终完成时间。对于流式问答来说，首字等待时间非常关键。论文里 TTFT P95 也作为正式指标报告。

153. 问：你的文档接入主结果是什么？

答：在 200 份正式样本上，全链路入库成功率是 100%，平均可检索就绪时延 2182.6 ms，P95 是 8983.0 ms。和旧基线相比，P95 降低了 99.09%。这说明系统不仅能跑通，而且入库链路已经明显优化。

154. 问：你的检索主结果是什么？

答：在 200 份文档、400 条查询上，正式配置取得 Recall@1=0.8675、Recall@5=0.9800、MRR@5=0.9051、nDCG@5=0.8562，检索 P95 为 1099.0 ms。这个结果说明系统在召回质量和在线时延之间达到了比较好的平衡。

155. 问：Recall@5=0.98 应该怎么解释？

答：可以理解为 400 条查询中，98% 的问题能在前 5 个结果里找到目标文档。这个指标说明系统“找得到”的能力很强。对企业问答来说，能把正确文档稳定召回到前几位，是后续回答可靠的前提。

156. 问：为什么你答辩里总强调 P95，而不是平均值？

答：因为平均值会把长尾问题冲淡，而系统实际可用性恰恰常常被长尾决定。P95 更能反映用户在大多数情况下是否会遇到明显卡顿。工程答辩里看 P95 比只看平均更有说服力。

157. 问：为什么精确匹配子集的 P95 反而比总体更高？

答：论文里已经解释过，这是因为该子集样本量更小、尾部波动更明显，不代表整体系统退化。也就是说，这个现象更多是统计上的尾部效应，而不是说明精确匹配一定更慢。答辩时不要回避，直接解释原因即可。

158. 问：30 组权限矩阵零越权，说明了什么？

答：说明当前权限规则在检索阶段已经真正生效，未授权文档没有被错误召回出来。它不是只测文件列表接口，而是直接测问答和检索边界。这个结果是“权限前置过滤有效”的核心证据。

159. 问：流式和历史相关测试的主结果是什么？

答：10 个会话、30 轮流式交互中，History API 和 History Context 都通过，流式完成率 100%，需要来源说明的锚点轮次引用标记通过率 100%，TTFT P95 为 1701 ms。说明系统不只会答，还能稳定流式返回并保持历史承接。

160. 问：为什么选择性历史策略的 0.7778 通过率是有意义的？

答：因为它显著高于无历史基线的 0.4444，也高于近轮拼接的 0.6889。更重要的是，它是唯一在长程历史回溯场景达到 1.0 的策略。也就是说，工作记忆和选择性历史确实改善了多轮语义稳定性。

161. 问：为什么近轮拼接在长程历史回溯上是 0？

答：因为固定窗口只能保留最近几轮，早期锚点信息会被挤掉。长程回溯恰恰依赖这些早期锚点，所以最近历史拼接在这类场景天然吃亏。这也说明“历史越近越好”并不总成立。

162. 问：检索模式对照实验说明了什么？

答：它说明不同检索路径各有优缺点：BM25 快但语义能力弱，Dense 语义更强但时延高，混合检索在精度和速度之间更均衡。论文不是为了证明“某一种绝对最好”，而是用实验说明为什么系统默认采用混合策略。

163. 问：为什么简单混合的 Recall@5 最高，但完整混合的 Recall@1 更高？

答：因为重排把高置信候选进一步前置了，所以首位命中率提升；但同时也可能把原本排在第 4、5 位的边缘正确结果挤出 top5。换句话说，重排在当前参数下优化了“第一名更准”，但牺牲了部分 top5 尾部召回。

164. 问：为什么完整混合会更慢？

答：因为它比简单混合多了一步重排计算，候选窗口内要做额外评分，所以时延上升是合理代价。论文里完整混合 P95 为 289 ms，明显高于简单混合的 140 ms。这个 trade-off 在答辩时要诚实表达。

165. 问：阶段热点分析说明了什么？

答：说明当前瓶颈不是 Kafka 调度，也不是 ES 写入，而是 embedding。Embedding 占平均总时延 71.47%，解析占 10.18%，ES 写入占 5.78%。所以后续优化重点应该放在 embedding 吞吐和长文档处理成本上。

166. 问：为什么“队列等待基本为 0 ms”很重要？

答：因为这说明引入 Kafka 之后并没有形成新的排队瓶颈。也就是说，异步化把长任务移出在线请求路径的同时，没有在当前规模下制造新的拥塞点。这个结论支撑了异步接入设计的合理性。

167. 问：并发实验的结论是什么？

答：在 50 到 1000 用户的五档梯度并发下，系统都没有失败请求；最高档 1000 用户、2000 请求时，P95 为 1191 ms，峰值 QPS 为 650.83，Hit@5 为 0.925。说明系统在当前单机条件下具备较好的扩展趋势和稳定性。

168. 问：为什么你说 P95 随并发增长呈亚线性？

答：因为用户数从 50 提高到 1000，增长了 20 倍，但 P95 从 161 ms 增到 1191 ms，只增长了大约 7.4 倍。说明系统没有随着并发数线性恶化。当然，这不代表没有瓶颈，而是说明在当前范围内还没有完全失控。

169. 问：隐私测试里的“零泄漏”是不是说明绝对安全？

答：不能这么说。更准确的表述是“在当前测试样本和测试范围内未观察到泄漏”。安全问题本身是开放的，所以论文里我不会把它说成绝对安全，而是说系统建立了有效的多层防线，并在当前测试中通过了验证。

170. 问：什么叫结构符合性检查？

答：就是检查系统的代码和配置是否满足论文宣称的模块边界，比如 Provider 解耦、服务分层、配置外置等。它不是功能测试，而是验证系统是否按照设计实现。对于系统型论文，这能证明实现和设计不是脱节的。

## 九、创新点、局限性与后续展望（171-185）

171. 问：你认为这篇论文最大的创新点是什么？

答：最大的创新点不是单个模块，而是把可靠接入、权限前置、会话编排和隐私治理整合成一套面向企业私有文档问答的统一架构。它把“文档怎么进来、怎么被授权召回、怎么在多轮里承接、怎么控制内容风险”连成了一条完整链路。这个系统级整合就是我最核心的贡献。

172. 问：你的创新更偏工程还是偏理论？

答：更偏工程机制和系统设计创新，而不是基础理论创新。我的工作关注的是在真实约束下如何把多个关键控制点组织起来，并证明这种组织是有效的。这个定位我会主动说清楚，而不是硬说成算法突破。

173. 问：如果只能挑一个贡献来讲，你会挑哪一个？

答：我会挑“检索前置权限过滤 + 会话编排”这一组，因为它最能体现企业场景和普通 RAG 系统的差异。很多系统能答，但不一定能在权限一致和多轮稳定的条件下答。我的系统就是把这两个最容易被忽略的约束真正落实到了检索和生成前。

174. 问：你觉得整套系统里最难实现的模块是什么？

答：从工程复杂度看，最难的是文档接入和多轮会话控制。前者要协调 MySQL、Redis、MinIO、Kafka 多层状态，后者要在不依赖大模型盲猜的前提下稳定判断上下文边界。这两个模块都不是“调个库就能解决”的问题。

175. 问：你觉得老师最可能重点追问哪个模块？

答：大概率是权限过滤、会话管理和隐私治理。因为这三块最能体现论文是不是只停留在“会问答”，还是确实考虑了企业场景的真实约束。尤其是“为什么不只靠历史拼接”和“为什么权限要前置”，通常都会被问到。

176. 问：你这篇论文最大的局限是什么？

答：最大的局限是实验规模和部署环境仍然有限，目前主要是在 200 份文档、400 条查询和单机受控环境下验证。它足以证明架构可行，但还不能直接外推到生产级规模。这个限制我在第六章里已经明确说明。

177. 问：为什么你不直接声称这是一个生产级系统？

答：因为从研究诚实性出发，当前系统还缺少生产级的可观测性、容错恢复和长期运行验证。它已经具备较强的工程原型能力，但距离真正长期稳定运行的企业生产系统还有差距。答辩时如实说明，比夸大更稳妥。

178. 问：为什么当前没有做多模态文档理解？

答：因为论文当前聚焦的是文本类企业文档的工程化 RAG，先把文本路径上的可靠接入、权限和会话问题跑通。多模态会把图像区域、表格单元和文本片段统一纳入权限和追溯体系，复杂度会大幅提高。它是合理的后续方向，但不适合在本论文里一口气做完。

179. 问：为什么没有做单元格级 Excel 语义建模？

答：因为当前版本优先解决的是“Excel 能不能进系统、能不能被问到、表格段能不能尽量保留”，而不是完整表格语义理解。单元格级建模需要额外的结构抽取和关系表达，会显著增加系统复杂度。论文当前把它明确作为后续增强方向。

180. 问：为什么没有做 chunk overlap 或层次化检索？

答：因为当前系统先选择了“语义边界切分 + topK 多证据组织”的相对简洁方案。Overlap 和层次化检索确实能进一步缓解跨块语义断裂，但也会增加索引规模、重复片段和实现复杂度。在本科论文里，我优先验证了基础架构的成立。

181. 问：为什么没有做更强、更系统的对抗安全 benchmark？

答：这确实是局限之一。当前论文已经覆盖了 45 条对抗提示和多类攻击样式，但还谈不上完整、标准化的安全 benchmark。后续需要建立更系统的分类评测和更大规模的对抗样本集。

182. 问：未来在评测方面最该补什么？

答：最该补的是更稳定、可复现的企业文档评测基准，包括更多文档类型、更复杂权限层级和更丰富的多轮场景。只有评测基准更强，后续的检索和会话优化才更有可比性。第六章里我也把这作为近期最迫切工作之一。

183. 问：未来在工程运维方面最该补什么？

答：最该补的是可观测性、故障定位和长期运行能力，比如链路指标、监控告警、任务补偿和索引维护工具。当前系统已经做了阶段性时延采集，但离完整运维体系还有差距。它是从“可行原型”走向“可运营系统”的关键一步。

184. 问：未来在检索层面你最想优化什么？

答：一是更稳的查询改写和重排策略，二是长文档与跨块信息的检索组织能力。特别是语义改写场景目前虽然 Recall 还可以，但排序质量还有提升空间。这个方向和系统当前的实验结果也是一致的。

185. 问：未来在多模态方面你认为最大的挑战是什么？

答：最大的挑战不是接一个图像模型，而是如何把图像区域、表格单元和文本片段统一纳入权限过滤和证据追溯体系。因为一旦进入企业场景，多模态也必须遵守同样的安全和授权规则。这个挑战是结构性的，而不只是模型接入问题。

## 十、高压追问与现场答法（186-200）

186. 问：如果老师说“你这不就是把现有技术拼起来吗”，你怎么答？

答：我会承认系统确实使用了现有成熟技术，但重点不在“用了哪些技术”，而在“如何围绕企业私有文档问答这个具体问题，把关键控制点组织起来”。现有技术并不会自动给出可靠接入、权限前置、会话承接和全链路隐私治理的统一方案。我的贡献是把这些环节系统化，并用实验验证它们确实带来了效果。

187. 问：如果老师说“创新性不够”，你怎么答？

答：我会说明这篇论文的创新类型是系统设计创新和工程机制创新，而不是基础算法创新。具体体现在四点：可恢复接入与异步入库、检索前置权限过滤、轮次分类驱动的会话控制、分阶段隐私治理。对本科毕业设计来说，这种围绕真实场景问题的系统性创新是成立的。

188. 问：如果老师说“你的指标不够领先”，你怎么答？

答：我不会和大规模公开 benchmark 去硬比，因为论文目标不是追 SOTA，而是在企业私有文档受控部署条件下验证架构是否有效。我的重点是效果、授权、安全和工程代价之间的平衡，而不是单一指标极致。正式实验已经证明这套设计在当前边界内是可行的。

189. 问：如果老师说“200 份文档、400 条查询还是太小”，你怎么答？

答：我会承认规模确实有限，这也是第六章明确写出的局限。但这个规模已经足以覆盖多格式文档、多类查询、多轮交互和权限安全验证，能够支持原型系统的可行性结论。它不能证明生产级上限，但能证明架构方向成立。

190. 问：如果老师说“你没有理论推导”，你怎么答？

答：我会说明这篇论文的研究对象是工程系统，不以数学推导为主要贡献形式。它的论证方式是需求分析、架构设计、实现细节和实验验证四位一体。对于系统型论文，这是一条合理的研究路径。

191. 问：如果老师问“为什么不用专门的向量数据库”，你怎么答？

答：因为我的系统需要的不只是向量近邻，还需要 BM25、重排和权限过滤在同一检索请求里协同工作。Elasticsearch 在这个问题上更符合混合检索需求。不是说向量数据库不好，而是当前系统目标更适合 ES。

192. 问：如果老师问“既然 BM25 这么快，为什么不直接纯 BM25”，你怎么答？

答：因为纯 BM25 在语义改写、同义表达和概念泛化场景下能力不够。论文里的对照实验也显示 BM25 的 Recall@5 明显低于混合方案。企业文档里很多问题不只是关键词匹配，所以不能只图快。

193. 问：如果老师问“既然 Dense 语义更强，为什么不直接纯向量检索”，你怎么答？

答：因为企业文档里还有大量术语、编号、金额和标题型查询，这些场景下词法信号仍然非常重要。纯向量可能语义相近，但未必对精确术语最稳。混合检索就是在这两类场景之间做平衡。

194. 问：如果老师问“为什么不直接微调一个大模型，让它自己学会这些知识”，你怎么答？

答：因为企业知识持续更新，频繁微调成本高、时效性差，而且权限边界也难通过参数微调精细控制。RAG 更适合处理动态知识、可追溯回答和授权范围控制。微调可以是后续增强，但不能替代当前系统的核心问题。

195. 问：如果老师质疑“隐私治理主要靠正则，会不会太弱”，你怎么答？

答：我会承认当前实现以规则和模式识别为主，确实不是语义级别的完美安全系统。但它的优势是可解释、可控、易于前置部署，适合作为第一道工程防线。后续完全可以结合更强的语义分类器或专门安全模型继续增强。

196. 问：如果老师问“零泄漏是不是绝对安全”，你怎么答？

答：我会明确说不是。更准确的表达是“在当前测试范围和样本内未观察到泄漏”。安全只能说“降低风险、提高防护能力”，不能轻易说绝对安全。

197. 问：如果老师说“你的多轮模块还是启发式规则，太 heuristic 了”，你怎么答？

答：我会承认它确实带有启发式特征，但这是刻意选择。因为在企业场景中，会话边界控制比完全交给大模型更需要可解释和可控。更重要的是，这套启发式编排已经在实验中证明比无历史和近轮拼接更稳定，所以它是有实证价值的。

198. 问：如果老师问“系统放大规模后最先出问题的会是哪一块”，你怎么答？

答：最可能先暴露问题的是 embedding 吞吐和长文档处理长尾，因为当前实验已经显示 embedding 是主瓶颈。再往后可能是 Elasticsearch 查询负载和多中间件协同下的可观测性问题。所以后续扩展应优先围绕这些位置优化。

199. 问：如果老师说“给你一分钟，只说一个最能证明你贡献的点”，你怎么答？

答：我会说，这篇论文最重要的不是“系统能回答”，而是“系统能在权限和安全约束下稳定地回答”。为此，我把文档可靠接入、检索前置权限过滤、轮次驱动的会话控制和全链路隐私治理连成了一条完整链路，并用正式测试证明它在当前边界内是有效的。

200. 问：如果老师最后让你用一分钟总结全文，你怎么答？

答：本文面向企业私有文档问答这一真实场景，针对可靠接入、授权一致、多轮承接和内容安全四类问题，设计并实现了一套工程化 RAG 系统。系统在架构上采用多存储协同、异步入库、混合检索、会话状态控制和分阶段隐私治理，在实现上完成了从上传、解析、切块、向量化到检索、Prompt、流式问答的全链路闭环。实验结果表明，在当前 200 份文档、400 条查询和多轮交互测试范围内，系统能够兼顾问答效果、授权边界和安全要求，因此验证了这套架构在企业私有文档场景下的可行性。
