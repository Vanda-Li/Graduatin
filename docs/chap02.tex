\section{业务场景分析}
\label{sec:business_scenario}

本系统服务于企业内部文档管理与知识问答场景。系统面向两类使用者：普通用户负责文档上传、检索与问答，操作范围限定于其授权组织；管理员在此基础上还承担组织标签的创建与层级维护、用户组织归属的配置以及共享边界的划定。

组织授权场景涉及管理员与普通用户两类操作主体，但两者的关注点并不相同。管理员侧的职责集中于组织标签的创建与层级维护、共享范围的划定，以及向具体用户分配可访问标签；这些配置共同决定系统的权限拓扑结构。普通用户登录后始终在一个当前主组织的上下文中操作。主组织切换不仅更新界面显示，还同步刷新文档归属判定、检索可见范围与文件访问权限，三者联动生效，不依赖重新登录。

文档接入场景中，普通用户在当前主组织上下文下发起上传请求。系统在接收文件之前完成格式类型、文件大小与内容合法性的校验，以拦截不符合入库条件的文件。传输完成后，文档解析、分块、向量化与检索索引写入在后台依次推进。上述步骤对大型 PDF 尤为耗时，若阻塞在请求线程中等待完成，上传响应将长时间挂起，网络中断还可能导致已完成的处理工作无法继续。因此，上传确认与后台入库须解耦，系统须独立维护各处理阶段的状态并提供查询接口，使用户随时获知文档是否已进入可检索状态。

知识消费场景涵盖检索与问答两类操作。用户以自然语言提问后，系统依据其当前主组织确定可见文档范围，在此范围内执行候选召回与排序，并基于命中片段生成带来源依据的回答，同时支持原文预览和授权文件下载。多轮追问是该场景的常见交互模式，后续问题往往省略前文实体的重复描述，需要系统结合对话历史理解指代关系。但历史上下文的引入不应放宽权限判定，每轮检索仍须独立执行权限过滤，避免上下文扩展带来越权召回。

\section{功能需求分析}
\label{sec:function_requirement}

根据第 2.1 节的三类业务场景，本文将功能需求划分为四个模块，模块间沿数据流方向单向依赖。用户与组织管理模块是执行前提，负责在运行时建立并维护权限上下文；文档接入与知识入库模块完成离线处理链路，为检索提供数据来源；检索与问答模块在已入库知识上执行候选召回与回答构造；会话交互与文档服务模块专注于结果交付与文件访问，不介入回答生成逻辑。

\subsection{用户与组织管理需求}

用户与组织管理需求涵盖身份管理与组织上下文维护。

\begin{enumerate}[label=\textbf{U\arabic*}]
    \item 系统应支持用户注册、登录、登出和当前用户信息查询；登录成功后应返回包含用户身份标识与当前主组织标签的认证凭据。
    \item 系统应在注册完成时自动为用户创建仅本人可见的私人组织标签，并将其设为默认主组织标签。
    \item 系统应支持管理员创建组织标签、维护标签层级并为用户配置访问范围；用户切换主组织后，后续操作的权限上下文应立即更新，不应依赖重新登录。
\end{enumerate}

\subsection{文档接入与知识入库需求}

文档接入属于离线后处理链路，单文件须经历解析、分块与向量化等串行阶段，耗时通常远高于在线请求；加之网络不稳定可能导致上传中止，系统须将上传接入与后台处理解耦，并向用户提供各阶段的可查询状态。

\begin{enumerate}[label=\textbf{D\arabic*}]
    \item 系统应支持 PDF、Word、Excel、TXT 等主流文档格式接入，并在接收上传前完成格式类型、文件大小与内容合法性校验。
    \item 系统应支持大文件断点续传；网络中断恢复后，传输从中断位置继续，合并完成后的文件须通过完整性校验。
    \item 文件上传完成后，系统应异步执行文档解析、向量化与入库，并向用户提供分阶段处理状态查询能力，直至文档标记为可检索。
    \item 系统应支持文档删除；删除完成后，关系型数据库、对象存储与向量索引中的关联数据须同步清除，不得残留孤儿记录。
\end{enumerate}

\subsection{检索与问答需求}

系统须同时满足召回完整性、召回精度与权限正确性三项约束，三者分别对应不同的业务压力。召回完整性要求候选集覆盖相关文档，不遗漏授权范围内的关键内容；召回精度在实体名称、编号与专有术语密集的查询场景中尤为重要，单纯的语义匹配因词义泛化容易降低定位精度，须结合精确匹配与候选排序加以补偿；权限正确性要求系统在候选集进入生成模型前完成过滤，已有研究指出越权召回是 RAG 系统的重要安全风险~\cite{ni2025trustworthyrag}。

\begin{enumerate}[label=\textbf{R\arabic*}]
    \item 系统应支持自然语言检索，返回与问题相关且可直接作为回答证据的候选片段。
    \item 系统应同时支持语义匹配与精确匹配，并对候选结果进行排序优化，以提高实体、编号与术语密集场景下的定位精度。
    \item 系统应在候选召回前执行权限过滤：依据用户身份和当前主组织确定其可访问的组织标签集合，仅保留归属于该集合或标记为公开的文档所生成的片段，未通过过滤的片段不得进入后续处理。
    \item 系统应基于已授权且与问题相关的候选片段生成带来源依据的回答；在多轮对话中，系统须以对话历史辅助理解后续追问，但每轮检索均须独立执行权限过滤，不因会话上下文扩展而放宽权限范围。
\end{enumerate}

该链路涉及查询输入、候选生成、权限过滤和回答构造，其用例关系如图~\ref{fig:retrieval-usecase} 所示。

\subsection{会话交互与文档服务需求}

会话交互与文档服务需求涉及问答结果的交付方式、对话历史的持久化管理以及文档的预览与下载操作。上述三类能力与回答生成模块解耦，各自承担独立职责。

\begin{enumerate}[label=\textbf{I\arabic*}]
    \item 系统应支持回答的实时增量呈现，并在回答结束后返回明确的完成状态。
    \item 系统应支持多轮对话历史的持久化保存、分页读取和按会话回读。
    \item 系统应允许用户在权限范围内查看文件列表、预览文本内容，并生成带时效性的安全下载链接。
\end{enumerate}

\section{非功能需求分析}
\label{sec:non_function_requirement}

非功能需求从性能效率、安全性、可靠性与可维护性四个维度加以讨论。由于本系统采用单节点部署、语料以中型 PDF 为主且并发规模有限，各维度的验收阈值需结合实际部署边界与验收对象设定。

\subsection{性能效率需求NF1}

NF1 分离线入库与在线交互两类时间指标分别规定验收要求。

可检索就绪时延定义为从文档上传完成至文档进入可检索状态的全链路耗时。目标语料以中型 PDF 为主（平均约 50 页，单文件不超过 20 MB）；结合单节点部署与后台异步处理链路的实际边界，验收要求设为均值不高于 $6\mathrm{s}$，P95 不高于 $10\mathrm{s}$。

在在线交互侧，检索响应时间 P95 不高于 $1500\mathrm{ms}$，$50$ 用户并发条件下的检索响应时间 P95 亦不高于 $1500\mathrm{ms}$。流式问答的首 token 时延（Time to First Token，TTFT）P95 不高于 $2.5\mathrm{s}$。

\subsection{安全性需求NF2}

为防止越权召回与敏感信息泄漏，安全性需求涵盖以下 3 项约束：
\begin{enumerate}[label=(\arabic*)]
    \item 所有非公开接口必须受身份认证保护；
    \item 检索阶段越权返回率应为 $0$；
    \item 问答输出中的原文泄漏率应为 $0$。
\end{enumerate}

其中，原文泄漏率定义为：在敏感信息问答样本中，回答中出现未授权文档的连续句段（长度不少于一句）或完整敏感字段值的问题数，占样本总数的比例。

\subsection{可靠性需求NF3}

NF3 关注上传恢复、后台处理和交互反馈三条最容易出现状态中断的链路。考虑到系统当前采用单节点部署，且文档处理同时涉及对象存储、消息队列与向量索引等多个组件，一旦任一环节失败便会直接影响文档可用性，因此验收重点放在恢复成功率、阶段成功率与状态一致性上。NF3 的验收要求如下：
\begin{enumerate}[label=(\arabic*)]
    \item 断点续传成功率不低于 $99.5\%$；
    \item 解析成功率、索引成功率和入库一致性通过率均不低于 $99.5\%$；
    \item 流式交互完成通知到达率不低于 $99.5\%$，历史记录接口应稳定返回并支持分页读取与连续追问。
\end{enumerate}

\subsection{可维护性与扩展性需求NF4}

可维护性与扩展性难以直接量化，本文以结构符合性通过率作为验收指标，依据系统预期演化边界逐项检查关键职责是否分离，共 6 项，全部通过则认为该维度达标。检查项如下：
\begin{enumerate}[label=(\arabic*)]
    \item 认证授权与业务处理分离；
    \item 上传接入与后台入库处理解耦；
    \item 检索召回与回答生成分层；
    \item 模型与检索关键参数外部化；
    \item 文件服务与会话管理职责独立；
    \item 外部模型服务具备替换边界。
\end{enumerate}

验收要求为上述 6 项全部通过，即结构符合性通过率达到 $100\%$。
