\chapter{需求分析}
\label{cha:requirement}

\section{系统建设目标}
\label{sec:requirement_goal}

本系统面向企业私有文档知识管理场景，目标是将分散在本地文件、办公文档和业务资料中的非结构化文本转化为可检索、可追溯、可问答的知识服务，集中体现在知识接入、权限治理和交互服务三个层面。首要目标是建立统一的文档接入与知识入库通道，使系统支持用户上传常见文档格式，并在上传完成后继续完成文件合并、文本解析、语义切分和向量索引构建，使原始文件稳定转化为后续检索可用的知识片段。

企业大模型设计研究指出，系统能力只有嵌入明确的组织边界与控制框架，才能转化为可治理的业务能力\cite{oleary2024enterprise}。因此另一个核心目标是建立面向企业组织结构的知识访问控制机制。系统不仅要完成用户登录认证，还需要将用户所属组织标签、私人空间和公开资源整合到同一权限模型中，确保不同角色在查询、下载、预览和删除文档时受到一致约束。权限语义必须贯穿上传、检索和文件服务全过程，而不能只停留在接口入口处，否则知识库越完善，越权访问带来的风险就越高。


第三项目标直接面向最终使用方式，即形成可交互的知识问答界面。系统需要允许用户通过自然语言提问触发检索与生成，并通过流式返回、多轮会话和来源引用改善企业内部知识获取体验。这里关注的不只是“能否回答”，还包括回答是否能够在组织边界内给出可核对的依据。

\section{业务场景分析}
\label{sec:business_scenario}

从业务角度看，系统涉及两类直接使用者：普通用户与管理员。普通用户主要负责上传个人或所属组织范围内的文档，并在权限范围内进行检索、问答、下载和预览；管理员除具备普通用户能力外，还承担组织标签维护、用户组织授权和系统级资源管理职责。系统整体业务流程如图 \ref{fig:requirement-overview} 所示，角色差异直接决定了功能边界，也决定了后续权限模型与检索过滤规则的设计方式。

\begin{figure}[H]
    \centering
    \includegraphics[width=0.82\textwidth]{image/chap02/rag.png}
    \caption{系统业务流程概览}
    \label{fig:requirement-overview}
\end{figure}

系统并非单一的检索工具，而是围绕“身份建立、文档接入、知识消费与文件服务”展开的连续业务过程。图~\ref{fig:requirement-overview} 左侧体现身份进入与文档接入，右侧体现知识消费与文件服务，两部分共同构成系统的完整业务闭环。

在账户与组织管理场景中，用户完成注册、登录后进入系统，管理员负责创建组织标签、设置父子标签关系，并为用户分配可访问的组织范围。文档接入环节则从知识库页面的上传动作开始，系统需要顺次完成分片传输、断点续传校验、服务端文件合并、文本解析、语义切分和向量索引构建。这意味着上传行为并不是孤立的文件提交操作，而是知识入库链路的起点。

当用户进入知识检索与问答场景后，系统需先依据用户身份与组织标签限制候选知识域，再根据检索结果构造提示词并触发大模型流式作答。与之相邻的文档服务场景负责承接结果消费过程，允许用户在已授权范围内查看文件列表、检索知识片段、预览文本内容、生成下载链接以及读取历史对话记录。只有将检索、问答与文件服务合并看待，才能准确界定系统在真实业务中的运行边界。对于内部规范与业务资料并存的文档集合，先完成稳定的组织与归属划分，再进入检索与问答流程，更符合机构规范文档研究揭示的建设路径\cite{normdocs2025}。

\section{功能需求分析}
\label{sec:function_requirement}

系统能力是沿着“身份建立、文档进入、知识检索、结果消费”逐步展开的。因此，本节按照用户实际操作链与系统内部依赖关系共同组织各项功能需求。

\subsection{用户与组织管理需求}

用户与组织管理模块承担系统的身份治理与权限配置职责，它决定了后续上传、检索、下载和问答所依赖的访问边界。对企业知识系统而言，这一模块的任务不是把账号信息单独保存下来，而是把用户与组织关系转化为后续各业务链路都能持续使用的权限上下文。

从具体功能看，系统首先需要提供用户注册、登录、登出和当前用户信息获取能力，管理员还需要能够创建组织标签、维护标签层级关系并向指定用户分配访问范围。与此同时，系统必须在注册时自动生成私人组织标签，并将其作为默认主组织标签，以保证用户首次进入系统后即可拥有独立私有空间；否则，即使认证过程已经完成，用户在首次使用时仍然缺少可直接落地的知识归属范围。

\subsection{文档接入与知识入库需求}

文档接入模块负责完成文件上传、断点续传和知识入库准备。该模块既是知识库内容增长的入口，也是后续解析、切分和索引构建的前置条件，因此其需求不能只停留在“文件能否传上来”这一层面，而要同时覆盖上传稳定性与入库可衔接性。

系统需要支持 PDF、Word、TXT 等常见文档格式，并在上传开始时完成基本的文件类型校验。对于大文件，系统还需要通过分片上传与断点续传机制在网络中断后恢复未完成分片，避免重复传输；全部分片上传完成后，服务端还应继续完成文件合并，并生成后续异步处理所需的任务信息。同时还需要将文档向量化存储，为后续检索提供基础。

\subsection{检索与问答需求}

检索与问答模块是系统的核心业务模块，因为企业用户最终感知到的“知识问答是否准确”，主要体现在这里。与单纯的文档检索不同，该模块还必须把召回结果转化为可供模型使用的上下文，因此其需求同时涉及检索质量、权限边界和生成约束。

系统需要支持用户输入自然语言问题，并完成与文档片段处于同一向量空间下的语义检索。对于专有名词、编号和术语密集场景，系统还应提供关键词匹配与向量检索结合的混合检索方式，以避免纯语义召回对精确术语的弱化。更关键的是，组织标签和文档可见性过滤必须在检索阶段完成，未授权文档不应进入最终召回结果，否则生成链路即使形式上保留了权限判断，实际仍可能在候选构造阶段引入越权信息。

图~\ref{fig:retrieval-usecase} 给出了本模块的用例图，其中检索结果还需要被组装为大模型外部知识上下文，并在无结果或向量生成失败时触发可控降级处理。这里的重点不只是找到相关内容，而是只把允许使用且足以支撑回答的内容交给生成链路。用户感知信息价值研究也表明，答案可接受性不仅取决于是否命中，还取决于来源感知和结果可信度\cite{llmvalue2025}。

\subsection{会话交互与文档服务需求}

除核心检索链路外，系统还需要提供会话管理与文档服务能力，以保证答案消费、历史回溯和文件操作能够形成完整闭环。这里承接的是用户对检索结果的持续使用过程，而不是单次提问后的即时展示。

系统需要支持基于 WebSocket 的流式回答，使用户能够实时看到模型生成内容，而不是等待完整答案一次性返回；系统还应支持对话历史保存与读取，以便在连续提问时保留必要上下文。围绕文件消费链路，系统应允许用户在权限范围内查看文件列表、预览文本内容和生成下载链接，并在回答中保留来源引用编号。相关功能关系如图~\ref{fig:interaction-usecase}所示，该图同时反映了会话交互与文档服务之间的耦合关系。若缺少这部分支撑，检索结果即使正确，也难以转化为可持续使用的知识服务。

\section{非功能需求分析}
\label{sec:non_function_requirement}

Dromey 的软件产品质量模型研究指出，性能、可靠性与可维护性等属性需要与软件产物的具体结构特征结合分析\cite{dromey1995quality}。Kitchenham 与 Pfleeger 的研究进一步说明，软件质量评价不能脱离实际产品行为与使用环境单独讨论\cite{kitchenham1996quality}。结合企业知识系统的运行特点，本文从性能效率、安全性、可靠性和可维护性四个方面提出非功能需求。

这些要求不直接决定系统是否能够完成某一项功能，却决定系统能否在持续运行、多人协同和真实业务压力下保持可接受的服务质量。对于知识问答系统而言，若非功能属性失守，即使功能项全部实现，也难以在真实环境中稳定部署。四类属性所约束的对象并不相同：性能关注链路时延与吞吐，安全关注知识边界，可靠性关注异常下的连续运行，可维护性则关系到系统后续迭代成本。

\subsection{性能效率需求}

系统面向文档上传与知识问答双重业务，需要兼顾离线入库性能与在线交互性能。在入库侧，系统应支持大文件分片传输与后台异步处理，避免在请求线程内完成全部解析与向量化工作，否则上传延迟会直接转化为用户等待成本。

在线问答侧的性能要求则更强调感知时延。系统应通过流式输出缩短首字等待时间，并在合理的候选窗口范围内完成检索与重排序，使用户能够在可接受时间内获得持续返回的结果，而不是长时间停留在无反馈状态。

\subsection{安全性需求}

系统处理的是企业私有知识，因此安全性是核心约束。系统应保证接口访问基于身份认证完成，文档访问受到角色、组织标签和公开属性的共同限制；系统应避免无权限用户在检索、预览、下载和删除等环节接触到未授权文档。与此同时，系统还应尽量减少敏感知识直接暴露给外部模型服务的范围，仅将必要片段作为上下文传递给生成模块，以控制知识在生成阶段的外送边界。

\subsection{可靠性需求}

系统应具备应对网络中断、接口波动和下游服务异常的基本能力。上传过程中应能够恢复中断分片，异步任务失败后应具备重试和死信处理机制，以保证文档从接入到入库的状态推进不会因为局部异常而中断。

当向量生成异常或检索执行失败时，系统还需要切换到可接受的降级路径，并保持用户可感知的状态反馈。可靠性的重点不在于完全消除异常，而在于使异常不会直接演变为核心功能整体不可用。

\subsection{可维护性与扩展性需求}

系统应采用前后端分离和分层设计，使用户界面、接口层、业务逻辑层和数据存储层保持清晰职责边界；同时，模型调用、向量生成和检索服务应通过配置与服务封装解耦，便于后续替换嵌入模型、对话模型或存储组件。随着知识规模增长，系统还应允许继续扩展文件类型、索引字段与检索策略，而不需要整体推翻既有实现。
