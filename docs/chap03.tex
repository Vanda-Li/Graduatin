\section{设计目标与总体思路}
\label{sec:design_goal}

企业私有文档知识服务同时面对三类工程压力：长文档接入后的持续处理、多租户场景下的权限约束，以及外部模型供应商可能发生的变更。三类压力叠加在同一同步请求链路上时，问题会相互放大：长文档解析与向量化一旦占用请求线程，在线响应延迟便随并发规模显著上升；权限判定与模型接入若与业务流程写死，则模型替换或权限规则调整时改动范围难以控制。

针对上述压力，本系统须回答三项具体问题：文档解析与向量化耗时不可预期，处理链路须从交互请求中脱离，进入可追踪的后台异步流程；权限判定须在检索与生成之前完成，使授权边界在候选召回、上下文装配和回答返回三个阶段保持一致；模型替换路径须局限在专用访问层，使问答编排与入库逻辑不随厂商变动而调整。

针对上述问题，系统在三个层面设置隔离边界：文档接入按“上传确认—异步调度—后台处理”组织，将长任务移出在线请求路径；权限过滤在候选召回之前执行，授权边界直接内嵌于检索阶段，而非在结果生成后再做后置裁剪；聊天生成与向量化均经由抽象提供者接口（Provider）访问，模型切换仅影响接口实现层。

\section{总体架构设计}
\label{sec:overall_design}

系统总体采用前后端分层结构，如图 \ref{fig:system-architecture} 所示。前端划分为知识库管理、聊天问答、组织标签管理和用户管理四个功能模块，负责文档接入、问题发起与结果展示；后端基于 Spring Boot，承担认证授权、文档处理、检索问答与会话管理。

持久化相关能力由 MySQL、MinIO 与 Elasticsearch 共同承担：MySQL 保存用户、组织标签与文件元数据，MinIO 存储原始文件对象，Elasticsearch 维护全文与向量检索索引。运行时支撑层由 Redis 与 Kafka 组成，前者缓存上传状态和会话信息，后者把文档接入链路与后台入库链路解耦，避免长任务滞留在交互路径中。模型层则以聊天 Provider 和向量 Provider 两个抽象对接外部模型，使上层业务不直接依赖具体厂商实现。

\section{核心模块设计}
\label{sec:core_design}

本节针对系统中最容易产生耦合或越权风险的三类边界展开设计说明：模型调用层的替换边界、权限与隐私的过滤边界，以及从文档入库到回答生成的主业务链路。

\subsection{模型 Provider 解耦设计}
\label{sec:model_provider_design}

本系统同时接入问答生成与向量化能力，二者在请求协议、吞吐特征与失败处理上存在显著差异；企业部署中，模型供应商还可能因成本与合规要求调整而变更。若上层业务直接耦合厂商接口，模型切换将导致问答编排、检索服务与文档入库链路同步改动，使后续模型替换和成本调整的代价显著上升。

RAGLab 强调 RAG 框架的模块化组织与研究透明性\cite{zhang2024raglab}；FlexRAG 通过统一配置管理和模块化装配支持灵活组合\cite{zhang2025flexrag}；RAGVA 则从工程实践角度讨论了 RAG 应用在构建、评测与维护中的挑战\cite{vogelsang2025ragva}。这些工作说明，RAG 系统的模块化组织和组件可替换性具有工程意义，可为本文隔离模型访问层提供参考；而 \texttt{ChatProvider} 与 \texttt{EmbeddingProvider} 的进一步拆分主要来自本系统自身的调用约束：生成调用以流式输出为主、对延迟更敏感，向量化调用以批量提交为主、对吞吐量要求更高，因此二者分别封装为独立的访问接口。

系统在业务层与模型调用层之间引入 \texttt{ChatProvider} 与 \texttt{EmbeddingProvider} 两个抽象接口，业务代码依赖接口而非具体实现，厂商适配逻辑集中在接口实现类中。会话编排层只消费流式生成、完成回调与模型标识等统一能力；向量化链路只依赖批量文本向量化接口。替换模型实现时，改动范围限定在对应的 Provider 实现类，问答编排与入库链路无需同步修改；缓存、批量处理等优化也可以在模型访问层内独立扩展，不影响上层业务逻辑。

\subsection{权限模型与多租户隔离设计}
\label{sec:permission_design}

企业文档访问需同时满足角色、组织归属与文档属性约束\cite{sandhu2000nist}。标准 RBAC 只以角色为判定依据；本系统在此基础上引入组织标签维度，使同一角色的用户仍可按组织归属细分访问范围。系统在用户侧保存角色、主组织标签与有效组织标签集合三项字段，在文档侧保存归属者与公开属性两项字段。主组织标签用于确定当前操作上下文与基础校验入口；文档的实际可见范围由有效组织标签集合展开后与文档归属者、公开属性共同判定。文档级可见性规则可写为：当 $\texttt{role}=\texttt{ADMIN}$ 时，用户可访问全部文档；当 $\texttt{role}=\texttt{USER}$ 时，其可见集合为
\[
\begin{aligned}
V(u)=&\{d \mid d.\texttt{owner}=u\}\cup\{d \mid d.\texttt{isPublic}=\texttt{true}\}\\
&\cup\{d \mid d.\texttt{orgTag}\in EffectiveTags(u)\}.
\end{aligned}
\]
其中，\texttt{EffectiveTags(u)} 表示用户在组织继承规则展开后的有效标签集合。为支持父级组织共享，系统需依据组织标签的父子关系与共享规则计算该集合；例如，用户属于“研发部-后端组”时，若“研发部”被标记为可继承，则其有效标签集合包含“后端组”与“研发部”两个标签。

图 \ref{fig:permission-mapping} 展示该映射关系。登录态仅提供身份验证，检索与文件服务阶段的过滤条件由可见组织范围、文档归属与公开属性共同构成。

\subsection{文档接入与知识入库链路设计}
\label{sec:document_pipeline_design}

文档接入链路将原始上传文件转换为可检索知识，整体流程见图~\ref{fig:document-pipeline}。文件在完成传输确认后进入后台处理阶段，随后依次完成文本解析、语义切分、向量化与索引写入。下文将对象传输阶段的 chunk 称为“分片”，将解析与索引阶段的语义单元称为“分块”，以区分文件传输与文本切分两类处理过程。

在处理路径的选择上，系统以文件大小为依据区分直通路径与异步路径：小文件在单次请求内完成合并与解析，无需经过队列；超过 1 MB 的文件进入“上传确认—异步调度—后台处理”流水线，将长任务与在线请求路径隔离。

无论进入哪条路径，文件在传输确认后都按统一的入库顺序处理：先解析原始内容，再执行语义切分、向量化和检索索引写入。上传阶段记录文件摘要、分片确认状态与合并结果；后台阶段在解析、向量化与索引写入之间保留可恢复的阶段状态，以便任务失败后从最近完成步骤继续处理。

解析阶段除提取原始文本外，还同步构建安全文本与归一化文本视图。切分策略以段落为基本语义单元：识别为表格段时按行组合切分，保留行内字段关系；普通段落超过 512 字符时依句子边界继续拆分；若单句仍超过上限，则借助 HanLP 分词进一步切分，确保分块长度始终落在向量化预算范围内。

分块上限设为 512 字符。目标语料以企业公告类文本为主，大多数自然段在该范围内仍可保持完整；同时，这一长度也能够把单次嵌入调用控制在当前部署模型的输入预算之内。由于段落、句子与表格行已提供显式语义边界，系统不设置固定字符重叠，以避免重复片段扩大索引规模并干扰后续重排序的评分稳定性。

\subsection{混合检索设计}
\label{sec:retrieval_design}

企业专业文档术语密集、语义边界模糊：纯向量检索在细粒度术语匹配上易出现偏移，纯关键词检索又难以覆盖同义表述。RAG 与 DPR 分别说明了稠密表示在检索增强问答和语义召回中的有效性\cite{lewis2020rag,karpukhin2020dpr}，但这类方法并不天然擅长处理术语、编号和金额等精确匹配问题；面向工程场景的对比研究进一步表明，将词法匹配叠加在稠密召回之后通常优于单一路径\cite{rahman2025comparative}。基于这一权衡，系统采用“权限前置过滤 + 向量候选召回 + 词法重排”的混合检索策略。

该流程分为两个阶段。第一阶段生成查询向量并执行 k近邻（k-nearest neighbors, kNN）召回，同时将用户身份、公开属性和组织标签作为过滤条件注入查询。第二阶段在候选窗口内叠加 BM25 文本匹配\cite{robertson2009bm25}，并利用 Elasticsearch 的 \texttt{rescore} 机制完成词法重排，以提升术语、编号与精确表述的排序稳定性。当前默认链路选择两阶段结构而不额外引入独立排序阶段，原因在于单节点部署下需要同时控制在线时延和实现复杂度。第~\ref{cha:system_test}~章将通过正式评测与消融实验验证这一取舍下的检索效果与响应时延。

为提升数字、金额、百分比及“万/亿”等表达的匹配稳定性，检索侧维护归一化文本视图，统一转换数值与单位，如“1万”转换为“10000”，“1亿”转换为“100000000”，“50\%”转换为“0.5”，使词法检索能够在更稳定的文本表示上执行。

在当前部署中，问答上下文默认取 Top-5，以控制送入生成模型的上下文长度。向量候选窗口按 30 倍放大，对应召回窗口为 150，用于在权限过滤后保留足够候选；\texttt{rescore} 窗口上限设为 75，用于给词法重排保留必要空间，同时限制在线开销。

\subsection{Prompt 工程设计}
\label{sec:prompt_design}

为避免系统指令、历史摘要与检索证据在同一上下文中相互干扰，Prompt 被设计为结构化载体，由统一装配器按固定逻辑槽位拼装。规则文案、会话策略、历史策略与隐私策略均外置为配置项，以便在不同部署环境下分别调整回答规则、历史注入方式与隐私约束。表 \ref{tab:prompt-layer-design} 展示其分层装配结构。

\subsection{会话状态控制与生成编排设计}
\label{sec:generation_design}

多轮对话中，追问、话题切换与元问题在检索策略上差异显著；若不加区分地将所有输入直接触发全局检索，短追问中的指代词很容易导致召回语义漂移。为此，系统在生成前先判定当前输入属于哪一类轮次，再决定是否引入历史、是否需要检索改写以及是否应收缩检索范围。轮次划分为 \texttt{NEW\_TOPIC}、\texttt{FOLLOW\_UP}、\texttt{META} 与 \texttt{CLARIFICATION} 四类，判定依据包括显式标记（新话题切换、历史引用）与上下文信号（短追问、长程历史回看、依赖上下文的问题模式）。具体路由关系如表 \ref{tab:turn-routing} 所示。

        \midrule
        \texttt{NEW\_TOPIC} & 显式新话题切换，或当前问题与既有上下文脱钩 & 重置工作记忆并允许全局检索 \\
        \texttt{FOLLOW\_UP} & 存在稳定回指对象的追问或延续性补充问题 & 沿用活跃文档集合和关键词收缩检索范围 \\
        \texttt{CLARIFICATION} & 输入过短且缺少明确实体，且最近历史中不存在稳定回指对象 & 暂停检索，先返回澄清请求 \\
        \texttt{META} & 询问系统行为、来源说明或会话状态等元问题 & 不触发知识库检索，转入会话控制分支 \\
        \bottomrule
    \end{tabularx}
    }
\end{table}

工作记忆仅保留多轮承接所需信息：当前主题与当前主语维持话题连续性，并为 \texttt{FOLLOW\_UP} 轮次的指代消解提供锚点；活跃文档集合与关键词在后续追问时收缩检索范围；最近独立问题与上一轮回答摘要则在历史注入时提供轻量上下文，以替代完整原始消息的传入。历史注入支持三种策略：\texttt{SELECTIVE} 从历史摘要中筛选与当前问题强相关的轮次注入，适用于会话较长且话题频繁切换的场景；\texttt{RAW\_RECENT} 保留最近若干轮原始消息，适用于对话较短且上下文连贯的快速追问场景；\texttt{DISABLED} 完全切断历史，适用于独立单轮检索或隐私要求较高的部署配置。第~\ref{cha:system_test}~章将对三种策略的效果进行对照比较。

轮次判断完成后，系统才调用混合检索服务获取候选片段，并把权限过滤后的结果组织为授权证据层，再与规则层、会话策略层、记忆策略层和工作记忆摘要共同装配为模型输入。来源编号随结果保留，用于回答后的证据回指；若轮次决策限定了文档范围或禁止全局回退检索，该约束同步作用于检索阶段。

\subsection{隐私治理设计}
\label{sec:privacy_design}

企业私有文档的敏感信息主要沿三条路径外泄：原文在入库阶段进入索引与检索层；未脱敏片段在 Prompt 装配与会话历史中进入模型输入；模型回答则可能在输出层重建敏感内容。结合企业场景下对访问控制与安全测评的要求\cite{gbt22239_2019}，三条路径分别对应接入、传递与生成三个环节，隐私治理须在每个环节独立设置控制点，而非依赖单一出口过滤。

基于风险传播路径，系统将治理对象划分为三类。个人身份信息（如身份证号、手机号）采用正则匹配后的掩码替换。系统凭据（如 API Key、访问令牌）命中后强制清除，不保留可复原片段。企业受限信息在脱敏后还需结合敏感级别判断是否允许外送；受限级别内容可在权限范围内参与内部检索，但不得进入外部 Embedding 计算、Prompt 上下文、长期历史保存和日志记录。

治理流程落在五个操作环节：
\begin{itemize}
    \item 识别：在文档解析与问题接收阶段标注敏感类型与级别。
    \item 脱敏：对命中内容执行掩码替换、占位替换或清除，生成 \texttt{safeText}。
    \item 标注：将受限标记、外部模型外送许可、敏感级别和敏感类型等元数据写入分块与索引记录。
    \item 约束：检索服务、Prompt 装配、会话历史与日志记录仅使用治理后内容。
    \item 复核：模型输出后再次扫描回答中的敏感模式；若命中高风险内容，则对结果执行拦截或再次脱敏。
\end{itemize}

图~\ref{fig:privacy-governance-chain} 给出了隐私治理在识别、脱敏、标注、外发与结果复核五个环节中的连续控制关系，以及索引视图、外发视图和日志视图三类下游使用分支。
