\section{系统架构设计}
本系统采用经典的 MVC（Model-View-Controller） 模式作为核心架构框架，并结合 RAG 架构范式 进行了针对性优化。业务架构从上至下划分为用户表现层、业务服务层和持久化数据层，实现了表现逻辑、业务调度与数据存储的深度解耦。

如图 3.1 所示，本系统业务架构以 AI/LLM Core 为核心驱动力，通过四个功能维度协同工作：

表现层作为人机交互的入口，遵循“系统状态可见性”与“流式反馈”的设计原则。负责渲染多轮对话界面、文档管理控制台以及权限配置看板。

业务服务层作为系统的核心引擎，采用模块化设计，划分为用户管理、文档处理、问题处理等功能模块。每个模块内部通过清晰的接口定义与数据契约实现高内聚、低耦合，支持独立开发与测试。

数据层负责系统中所有结构化与非结构化资产的存储与检索支撑。其中，关系型数据库（如MySQL）用于存储用户账户信息、权限配置与文档元数据；向量数据库（如Pinecone）则专门用于存储文档的语义向量表示，以支持高效的语义检索。文件存储存储文档原件，确保数据的完整性与可追溯性。

\subsection{前端展示层}
本系统前端采用响应式设计与多端适配策略，通过媒体查询与弹性布局架构，确保了系统在移动端与个人计算机端的跨平台一致性体验。相比于开发独立的移动端 App，采用响应式多端适配策略能够大幅降低代码维护成本，并在不损失核心 RAG 交互功能的前提下，实现企业移动办公场景与深度桌面办公场景的无缝切换。
\subsection{API与安全层}
系统是前后端完全分离的架构模式，通过标准化的通信协议实现表现层与业务层的交互。 
\begin{enumerate}
	\item 本系统采用Restful API设计原则，相比于 SOAP 等传统协议，RESTful 架构具备更好的轻量化特性与无状态性（Stateless），能够有效降低服务器端开销，并显著提升接口在不同编程语言间的可维护性与互操作性。
	\item 本系统采用Spring WebSocket实现流式通信，WebSocket 建立的长连接支持全双工通信，能以极低的延迟将大模型生成的 Token 实时推送至前端，确保了对话过程的连贯性。。
	\item 本系统使用 JWT 技术，通过在客户端存储加密 Token，实现了无状态鉴权，不仅减轻了服务器负担，且天然支持跨域访问（CORS），增强了系统在云原生环境下的伸缩性
	\item 本系统利用spring security声明式安全特性，将用户认证与组织标签鉴权逻辑进行深度集成，确保了每个 API 接口的访问均受到严格的 RBAC（基于角色的访问控制）保护，从根本上防止了接口级别的越权攻击。
\end{enumerate}
\subsection{业务逻辑层}
本系统后端基于 Spring Boot 框架构建，采用微服务化解耦思想将业务逻辑划分为用户管理、文档上传与处理、知识库检索及对话交互四个核心模块。
\subsubsection{用户管理模块}
本模块采用 Spring Security 构建安全核心，利用其高度可定制的过滤器链实现基于角色访问控制。区别于传统的单一权限模型，本系统在 Spring Security 的权限基础上，扩展了基于 JWT Claims 的组织标签机制，此方案通过解析 JWT 载荷中的标签，在服务层执行非侵入式的多租户数据过滤，显著降低了越权访问的风险。
\subsubsection{文档上传与处理模块}
文档处理模块通过分治策略与异步架构确保海量异构文档的入库效率。通过前端切片与后端 MinIO composeObject 接口。相较于传统的文件流写入，MinIO 的原子合并能力保障了分布式环境下文件合并的强一致性。引入 Redis BitSet 存储分片索引，维护文件断点续传状态。同时RAG 系统中，文档解析与向量化属于典型的计算密集型任务。通过 Kafka 缓冲请求，可实现流量削峰填谷，并通过消费者组水平扩展解析节点，确保系统具备高吞吐能力。
\subsubsection{知识库检索模块}
本模块采用混合检索策略，系统集成 Elasticsearch 的稠密向量检索与基于 IK 分词器的全文倒排索引检索（BM25）。相比于单一的稠密向量检索，混合检索能够在保证语义匹配的同时，兼顾关键词匹配的精确性，显著提升了检索结果的相关性与覆盖率\cite{Sawarkar_2024}。同时利用 Redis 缓存高频检索路径，可以有效缓解向量检索对 CPU 的重复计算压力，显著降低系统的平均检索耗时。
\subsubsection{对话交互模块}
前端会话采用 WebSocket 实现全双工通信，支撑流式生成。在此基础上，系统在多轮对话中，通过 Redis 维护对话历史与上下文状态，确保了用户提问的语义连贯性。整体前端采用Vue 3 + TypeScript 构建，利用其组件化与响应式特性，实现了对话界面、文档管理控制台与权限配置看板的高效开发与维护。通过Axios库与后端 RESTful API 进行数据交互，确保了前后端的高效通信与数据一致性。

\subsection{数据访问层}
通过 Spring Data JPA 结合 Hibernate ORM 框架实现数据访问层设计，采用 Repository 模式封装底层数据操作，通过抽象接口屏蔽具体的 SQL 编写细节。Hibernate 提供的二级缓存机制与延迟加载特性，可以有效减少数据库 I/O 次数
\subsection{数据存储层}
数据存储层适应前述要求，采用MinIO文档存储，MySQL数据库，Elasticsearch 向量数据库和Redis缓存数据库四种不同类型的数据库系统，分别满足文档原件存储、结构化数据存储、向量化存储与高频访问数据的快速读写需求。MySQL相比于 PostgreSQL 在高并发读取场景下具备更成熟的索引优化与较低的运维复杂度。MinIO 相比于传统的本地文件系统存储，兼容 S3 协议，具备更好的水平扩展性与冗余纠错能力。Elasticsearch 提供的 HNSW 索引算法 能够实现亚秒级的稠密向量检索，同时利用倒排索引保障了专业术语检索的召回率,维护成本也比 Milvus 等专门的向量数据库更低。Redis 作为内存数据库，提供了极高的读写性能，是当前工业界主流的高性能缓存数据库。

\section{编译环境配置}

编译环境配置相对来说比较简单，下载Tex Live2020并如同一般的程序一样安装即可。

\subsection{编译环境配置：Window篇}

在\url{https://mirrors.tuna.tsinghua.edu.cn/CTAN/systems/texlive/Images/}上下载Tex Live2020并参考教程\footnote{可以参考\url{https://zhuanlan.zhihu.com/p/58811994}}安装即可。

\subsection{编译环境配置：Linux篇}

在\url{https://mirrors.tuna.tsinghua.edu.cn/CTAN/systems/texlive/Images/}上下载Tex Live2020并参考教程\footnote{可以参考\url{https://zhuanlan.zhihu.com/p/55894177}}安装即可。


\subsection{编译环境配置：MacOS篇}

在MacOS上配置Latex的环境，这里我们使用的是MacTex。

\begin{enumerate}
	\item \url{https://www.tug.org/mactex/}下载MacTex安装。
	\item 安装步骤：不详细展开，按照图形界面点击即可, 傻瓜式安装。
\end{enumerate}

TIPS：MacTex文件比较大，有2G多，介意的话可以选择MacTex\_Basic包，只有100M以内，但是如果安装MacTex\_Basic，后期可能会遇到各种缺包的问题。


安装完成之后，可以简单测试一下安装是否成功。如可以查看Texshop应用是否安装好，或者在命令行测试一下\texttt{xelatex}命令是否可用。

\section{写作环境配置}

不同的写作工具对应不同的写作环境。这里我们给出几个工具的配置例子以供参考。

\subsection{模板编译流程}

由于\LaTeX 的限制，本模板需要经过四次编译才能生成完整的论文：

\begin{enumerate}
	\item 先使用xelatex编译一次
	\item 再使用bibtex编译一次
	\item 然后使用xelatex编译两次
\end{enumerate}

本编译流程已经写在Makefile中，修改模板源码后只需要执行\texttt{make main}即可按照该流程进行编译并生成最终的pdf。

\section{如何开始写毕业论文（设计）}

首先将所有个人信息，包括学号、姓名、专业、论文题目等，在\texttt{./docs/info.tex}中逐项进行更新。

然后我们再编辑\texttt{./docs/abstract.tex}补充论文摘要。

到了论文主体部分，我们可以自行编辑\texttt{./docs/chap01.tex}，\texttt{./docs/chap02.tex}等文件进行编辑。如果章数不够，可以自行修改\texttt{main.tex}增加新的章节。

当论文主体编写完成后，我们再编辑\texttt{./docs/ack.tex}作为论文致谢。
