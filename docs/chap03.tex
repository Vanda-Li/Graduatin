\section{系统架构设计}
本系统采用经典的 MVC（Model-View-Controller） 模式作为核心架构框架，并结合 RAG 架构范式 进行了针对性优化。业务架构从上至下划分为用户表现层、业务服务层和持久化数据层，实现了表现逻辑、业务调度与数据存储的深度解耦。

表现层作为人机交互的入口，遵循“系统状态可见性”与“流式反馈”的设计原则。负责渲染多轮对话界面、文档管理控制台以及权限配置看板。

业务服务层作为系统的核心引擎，采用模块化设计，划分为用户管理、文档处理、问题处理等功能模块。每个模块内部通过清晰的接口定义与数据契约实现高内聚、低耦合，支持独立开发与测试。

数据层负责系统中所有结构化与非结构化资产的存储与检索支撑。其中，关系型数据库（如MySQL）用于存储用户账户信息、权限配置与文档元数据；向量数据库（如Pinecone）则专门用于存储文档的语义向量表示，以支持高效的语义检索。文件存储存储文档原件，确保数据的完整性与可追溯性。

\subsection{前端展示层}
本系统前端采用响应式设计与多端适配策略，通过媒体查询与弹性布局架构，确保了系统在移动端与个人计算机端的跨平台一致性体验。相比于开发独立的移动端 App，采用响应式多端适配策略能够大幅降低代码维护成本，并在不损失核心 RAG 交互功能的前提下，实现企业移动办公场景与深度桌面办公场景的无缝切换。
\subsection{API与安全层}
系统是前后端完全分离的架构模式，通过标准化的通信协议实现表现层与业务层的交互。 
\begin{enumerate}
	\item 本系统采用Restful API设计原则，相比于 SOAP 等传统协议，RESTful 架构具备更好的轻量化特性与无状态性（Stateless），能够有效降低服务器端开销，并显著提升接口在不同编程语言间的可维护性与互操作性。
	\item 本系统采用Spring WebSocket实现流式通信，WebSocket 建立的长连接支持全双工通信，能以极低的延迟将大模型生成的 Token 实时推送至前端，确保了对话过程的连贯性。。
	\item 本系统使用 JWT 技术，通过在客户端存储加密 Token，实现了无状态鉴权，不仅减轻了服务器负担，且天然支持跨域访问（CORS），增强了系统在云原生环境下的伸缩性
	\item 本系统利用spring security声明式安全特性，将用户认证与组织标签鉴权逻辑进行深度集成，确保了每个 API 接口的访问均受到严格的 RBAC（基于角色的访问控制）保护，从根本上防止了接口级别的越权攻击。
\end{enumerate}
\subsection{业务逻辑层}
本系统后端基于 Spring Boot 框架构建，采用微服务化解耦思想将业务逻辑划分为用户管理、文档上传与处理、知识库检索及对话交互四个核心模块。
\subsubsection{用户管理模块}
本模块采用 Spring Security 构建安全核心，利用其高度可定制的过滤器链实现基于角色访问控制。区别于传统的单一权限模型，本系统在 Spring Security 的权限基础上，扩展了基于 JWT Claims 的组织标签机制，此方案通过解析 JWT 载荷中的标签，在服务层执行非侵入式的多租户数据过滤，显著降低了越权访问的风险。
\subsubsection{文档上传与处理模块}
文档处理模块通过分治策略与异步架构确保海量异构文档的入库效率。通过前端切片与后端 MinIO composeObject 接口。相较于传统的文件流写入，MinIO 的原子合并能力保障了分布式环境下文件合并的强一致性。引入 Redis BitSet 存储分片索引，维护文件断点续传状态。同时RAG 系统中，文档解析与向量化属于典型的计算密集型任务。通过 Kafka 缓冲请求，可实现流量削峰填谷，并通过消费者组水平扩展解析节点，确保系统具备高吞吐能力。
\subsubsection{知识库检索模块}
本模块采用混合检索策略，系统集成 Elasticsearch 的稠密向量检索与基于 IK 分词器的全文倒排索引检索（BM25）。相比于单一的稠密向量检索，混合检索能够在保证语义匹配的同时，兼顾关键词匹配的精确性，显著提升了检索结果的相关性与覆盖率\cite{Sawarkar_2024}。同时利用 Redis 缓存高频检索路径，可以有效缓解向量检索对 CPU 的重复计算压力，显著降低系统的平均检索耗时。
\subsubsection{对话交互模块}
前端会话采用 WebSocket 实现全双工通信，支撑流式生成。在此基础上，系统在多轮对话中，通过 Redis 维护对话历史与上下文状态，确保了用户提问的语义连贯性。整体前端采用Vue 3 + TypeScript 构建，利用其组件化与响应式特性，实现了对话界面、文档管理控制台与权限配置看板的高效开发与维护。通过Axios库与后端 RESTful API 进行数据交互，确保了前后端的高效通信与数据一致性。

\subsection{数据访问层}
通过 Spring Data JPA 结合 Hibernate ORM 框架实现数据访问层设计，采用 Repository 模式封装底层数据操作，通过抽象接口屏蔽具体的 SQL 编写细节。Hibernate 提供的二级缓存机制与延迟加载特性，可以有效减少数据库 I/O 次数
\subsection{数据存储层}
数据存储层适应前述要求，采用MinIO文档存储，MySQL数据库，Elasticsearch 向量数据库和Redis缓存数据库四种不同类型的数据库系统，分别满足文档原件存储、结构化数据存储、向量化存储与高频访问数据的快速读写需求。MySQL相比于 PostgreSQL 在高并发读取场景下具备更成熟的索引优化与较低的运维复杂度。MinIO 相比于传统的本地文件系统存储，兼容 S3 协议，具备更好的水平扩展性与冗余纠错能力。Elasticsearch 提供的 HNSW 索引算法 能够实现亚秒级的稠密向量检索，同时利用倒排索引保障了专业术语检索的召回率,维护成本也比 Milvus 等专门的向量数据库更低。Redis 作为内存数据库，提供了极高的读写性能，是当前工业界主流的高性能缓存数据库。

\subsection{文档解析与向量化组件}
文档管理与处理组件（如图3.5所示）是本系统的核心数据工厂，负责文档从原始状态到高维向量的生命周期管理。该组件通过解耦上传、解析、分块与向量化四个阶段，构建了高性能的知识入库流水线。

采用 Redis BitSet 维护分片上传状态位图。前端（基于 React Dropzone 或 Fine Uploader）将大文件切分为分片，上传前通过 MD5 算法生成全局唯一标识。后端通过对比 Redis 状态位，实现断点识别。

后端接入 MinIO 对象存储系统。在确认所有分片完整性及顺序后，调用其 composeObject API 执行原子合并。相较于传统文件流合并，该方案具备更强的操作原子性与回滚能力，合并完成后自动执行碎片清理，释放系统空间。

系统集成 Apache Tika 作为通用解析网关，针对非结构化数据，系统在解析过程中执行语义分块，按照预设长度对文本进行重构，为后续向量检索提供颗粒度适中的信息单元。

为解决文档解析（CPU 密集型）与大模型向量化（I/O 与 API 限制型）之间的性能极差，系统引入 Apache Kafka 实现了全链路的异步化解耦。解析完成的文档分块被发布到 Kafka 主题，Kafka 作为高性能消息总线，承担了流量削峰填谷的作用。通过消息有序性配置，确保同一文档的不同分块任务按顺序处理，并支持失败后的自动重试与任务状态实时监控。向量化服务作为消费者异步订阅并处理这些分块，最终将生成的向量持久化至 Elasticsearch 向量数据库中。

\subsection{智能检索与安全过滤组件}

为确保检索结果的精确度并兼顾企业级数据安全，本研究设计了一套集成身份感知与双路重排的混合检索体系（如图 3.6 所示）。
\begin{figure}[h]
	\centering
	\includegraphics[width=1.0\textwidth]{image/chap03/retrive.png}
	\caption{智能检索与安全过滤序列图}
	\label{fig:image-embedding-text}
\end{figure}

系统在检索入口处引入了基于 JWT 的身份拦截机制，实时提取查询主体的主组织标签与所属组织标签。然后将用户的查询向量化，并执行语义匹配。关于语义匹配本系统采用 k-NN 算法执行初步的语义海选。所有的向量检索请求在触达索引分片前，必须穿过一个前置权限过滤闸门。该闸门将组织标签作为强制性谓词注入检索算子，只有权限声明范围内的文档向量才被允许参与余弦相似度计算，未授权文档在检索初级阶段即被剔除。语义匹配的度量准则采用余弦相似度（Cosine Similarity），其数学定义如下：

$$Similarity(V_q, V_d) = \frac{\sum_{i=1}^{n} q_i d_i}{\sqrt{\sum_{i=1}^{n} q_i^2} \cdot \sqrt{\sum_{i=1}^{n} d_i^2}}$$
其中，$V_q$ 与 $V_d$ 分别代表查询向量与文档分块向量。通过 k-NN 搜索，系统可从海量异构分块中快速召回语义关联度最高的候选集。

针对第一阶段召回的候选文档集，系统启动第二阶段的关键词筛选和重排序（Rescore）。该阶段旨在弥补向量检索在专业术语及缩略词匹配上的“意图偏移”缺陷，保证最终结果的聚焦性。系统采用 BM25 算法 对候选文档的文本分量进行词频相关性评估，其计算公式为：$$Score(D, Q) = \sum_{q_i \in Q} IDF(q_i) \cdot \frac{f(q_i, D) \cdot (k_1 + 1)}{f(q_i, D) + k_1 \cdot (1 - b + b \cdot \frac{|D|}{avgdl})}$$通过该算法，系统能够精确捕获用户查询中的高价值关键词，确保检索结果在语义相近的前提下，关键术语的命中率达到最优。

为综合评价文档的最终价值，系统引入了线性加权融合排序模型。该模型通过对语义分与关键词分进行标准化加权求和，生成最终的排序度量（Final Score）：$$Final\_Score = \alpha \cdot Score_{bm25} + \beta \cdot Score_{cosine}$$式中，$\alpha$ 和 $\beta$ 为权值系数。基于实验评估与 RAG 应用对知识准确性的严谨要求，本研究采用了“侧重文字精确匹配，兼顾语义联想”的参数策略，设定 $\alpha = 1.0$，$\beta = 0.2$。这种非均衡权重分配方案确保了检索结果在专业语境下的严谨性，同时利用语义分作为排序微调因子，在关键词命中程度相近时，优先推荐语义逻辑更契合用户意图的内容。

\subsubsection{动态编排与上下文压缩}
为解决 LLM 长度限制与推理成本问题，本组件设计了启发式上下文编排算法。使用定界符 <<REF>> 与 <<END>> 构建“语义栅栏”，使模型能够精准区分“外部参考资料”与“自身参数知识”，防止指令篡改。为平衡 Token 成本与信息密度，单段知识片段字数限制在 300 字以内，并根据剩余上下文窗口自动调整历史对话的装载深度。当检索结果为空时，自动注入“（本轮无检索结果）”占位符，强制模型在无证据时不进行臆造。

\subsection{生成决策参数配置}

为确保大语言模型在处理知识库问答时能够输出准确、合规且具备溯源性的内容，系统对生成过程中的核心参数进行了精细化配置。这些参数不仅控制了模型的生成多样性，还通过结构化约束降低了模型产生幻觉或被指令注入攻击的风险。
生成决策由采样参数矩阵 $P = \{T, P_k, M\}$ 控制。
其中
\begin{enumerate}
	\item 温度系数 ($T$)：针对事实问答场景，设定 $T \in [0.2, 0.3]$ 以压制随机性；针对创意场景设定 $T > 0.7$。
	\item 核采样 ($P_k$)：常设为 0.8–0.95，配合温度系数确保输出的稳定性。
	\item 最大令牌数 ($M$)：限制单次生成长度，防止长尾输出浪费 Token。
\end{enumerate}
