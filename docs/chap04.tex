\section{身份认证与组织权限实现}
\label{sec:auth_impl}

认证链路在令牌快照与实时查库之间存在取舍：前者可降低查库频率，但权限变更无法即时生效；后者可保证实时性，但对高频请求的认证开销不可忽视。为此，系统将身份认证与组织授权拆分为两层独立过滤器，通过 Spring Security 过滤链依次执行，其调用关系见图~\ref{fig:security-filter-chain}。

\begin{figure}[!htbp]
    \centering
    \includegraphics[width=0.88\textwidth]{image/chap04/fig4-1-security-filter-chain.pdf}
    \caption{安全过滤链执行关系图}
    \label{fig:security-filter-chain}
\end{figure}

两层过滤器只共享身份，不共享授权结论。令牌解析阶段将用户身份写入 \texttt{SecurityContext}，组织边界校验再基于该身份独立完成，从而把身份恢复与资源授权明确拆开。

\subsection{令牌认证链路实现}

JSON Web Token（JWT）载荷携带 \texttt{role}、\texttt{userId}、\texttt{orgTags} 和 \texttt{primaryOrg}。字段 \texttt{orgTags} 记录令牌签发时的组织归属快照，仅用于快速恢复用户身份，不作为实时授权依据。过滤器在单次解析中完成用户上下文恢复并写入 \texttt{SecurityContext}；后续资源判定则转交 \texttt{OrgTagCacheService} 提供的实时有效标签集合，管理员调整组织权限后，无需等待令牌重签即可生效。


\subsection{私人空间与组织标签实现}

注册流程在创建用户记录后即按 \texttt{PRIVATE\_TAG\_PREFIX + username} 规则生成私人标签并将其设为默认主组织，随后依次回写 \texttt{orgTags} 与 \texttt{primaryOrg} 字段并刷新缓存，确保组织归属在用户首次登录前已完成初始化。

缓存层维护直接标签集合与有效标签集合。前者记录用户直接归属的组织节点，后者沿父级关系递归展开，覆盖所有可继承的上层组织。检索过滤与文件列表权限均基于有效标签集合完成过滤（见图~\ref{fig:org-filter-flow}），层级继承的递归展开因而收敛在缓存层，业务查询阶段无需反复遍历组织树。

\section{文档上传与异步入库实现}
\label{sec:upload_impl}

大文件接入跨越上传、合并和后台处理多个阶段，任何一处状态失配都会影响后续入库。上传环节需要避免重复写入，合并环节要校验分片状态与对象实体的一致性，后台处理则必须支持失败后的重新进入。

\subsection{前端分片上传与断点续传}

前端以 \texttt{useKnowledgeBaseStore} 维护上传任务队列，由 \texttt{calculateMD5} 以 5~MB 为粒度增量读取文件切片并累积计算哈希，避免大文件整体加载带来的浏览器内存压力。上传接口以 \texttt{fileMd5}、\texttt{chunkIndex}、\texttt{chunkMd5} 及二进制分片为载荷，服务端在响应中返回已确认的 \texttt{uploadedChunks} 列表；续传时，前端据此仅补发缺失分片。

\texttt{UploadController} 在首个分片到达时同步校验文件扩展名与多用途互联网邮件扩展类型（Multipurpose Internet Mail Extensions, MIME），拒绝扩展名和实际类型不符的文件。分片发送失败后，任务立即停止自动重试并转为人工续传，以免网络抖动期间静默产生重复请求，打乱服务端已确认分片计数与前端进度显示。服务端再借助 Redis 位图实现分片级幂等，同一分片重复到达时直接返回已确认状态。

\subsection{服务端分片落盘与对象合并}

服务端分片落盘与对象合并需要协调 MySQL、Redis 和 MinIO 三层状态。上传接口先确保 \texttt{file\_upload} 中存在文件记录，再依据 Redis 位图判断分片是否已上传，最后将新分片写入 MinIO 并补写分片记录。

上传状态由 MySQL、Redis 和 MinIO 共同维护。三层状态的划分依据访问特征而定：分片记录与文件元数据涉及事务性写入，由 MySQL 承担持久化；重传探测需要毫秒级响应且无需持久，适合 Redis 位图做幂等判断；MinIO 作为对象存储不适合处理高频小粒度写入，仅保存合并后的对象内容。

分片按“MinIO 写入 $\to$ Redis 位图更新 $\to$ MySQL 记录写入”的顺序推进。MinIO 对象覆盖写天然满足幂等性，因此位图更新前若发生失败，重传只会覆盖原对象，不会引入重复数据。MySQL 记录缺失时，后续合并会因记录数与分片对象数不一致而被拒绝；前端随后通过 \texttt{/upload/status} 重新拉取确认列表，仅补传异常分片后再重试。

全部分片到齐后，前端调用 \texttt{/api/v1/upload/merge} 触发合并，请求载荷包含 \texttt{fileMd5} 与文件名。合并阶段先根据 \texttt{file\_upload.totalSize} 计算期望分片数，再核对 \texttt{ChunkInfo} 记录数量，并逐个检查 MinIO 中的分片对象是否存在；任一步不一致都直接拒绝合并。若合并被拒绝，前端通过 \texttt{/upload/status} 重新获取 \texttt{uploadedChunks}，只补传缺失分片后再重试。文件合并流程见图~\ref{fig:file-merge-flow}。

合并前先校验分片记录数与对象存在性，把状态偏差阻断在 \texttt{composeObject} 调用之前。当前实现以串行方式逐一调用 \texttt{statObject} 检查分片存在性；分片数量较多时会引入线性等待，该路径的并行化留待后续优化。

对象合并成功后，系统立即删除上传标记，并把 \texttt{file\_upload.status} 推进为已合并，为后续入库提供稳定起点。

\subsection{Kafka 异步入库实现}

异步入库链路通过 Kafka 解耦上传确认与后台处理，使上传接口不再等待完整入库结束。合并完成后，控制器依据文件大小分流：小于 \texttt{file.processing.inline-size-threshold}（默认 1 MB）的文件同步执行入库逻辑，以减少小文件进入消息队列后的排队延迟；其余文件投递至 Kafka，由后台异步处理。

为保证处理链路的一致性，系统采用“状态先行”策略：文件状态先写入“已合并”，再发送处理消息；若消息投递失败，由重试机制与死信主题承担补偿。\texttt{KafkaConfig} 负责处理主题、死信主题和监听并发度等配置，监听器通过 \texttt{DefaultErrorHandler} 与指数退避策略处理失败消息。分区键使用文件 MD5，使同一文件的任务始终落在同一分区。

消费者收到任务后，优先按 \texttt{merged/fileMd5} 从 MinIO 下载对象；路径不可用时再回退到任务中的文件路径或 URL。下载成功后，\texttt{FileProcessingConsumer} 继续调用解析与向量化服务；处理失败时抛出异常，由监听器重试，重试耗尽后写入死信主题。

\subsection{文档解析实现}

文档解析需在内存受限条件下处理任意大小的文件，同时尽量保持语义切分边界的完整性。纯文本文件可直接进入解析流程，以减少额外的格式识别开销；其余格式则通过 Tika 的 \texttt{AutoDetectParser} 进行流式文本提取。

为防止大文件整体加载，系统采用“父块缓冲、子块切分”的两级处理策略：\texttt{StreamingContentHandler} 在 SAX 回调中持续累积字符，当缓冲量达到父块阈值时触发切分，将父块交由语义切分器拆解为子块；句子超过长度限制时，再由 HanLP 在句界处继续细分，保证最终语义单元的粒度可控。

文本提取完成后，系统以 \texttt{file.parsing.chunk-size} 控制最终子块长度。切分逻辑由 \texttt{ParseService} 按类型分支实现：表格段保留行结构写入单独分块；普通超长段落调用句界检测继续拆分；单句仍超长时由 HanLP 的分词结果提供断点，确保最终分块长度受控。每个子块入库前执行 \texttt{protectForIndexing}，同步写入原始文本、安全文本、归一化文本以及敏感级别、受限标记和敏感类型，并在文档结束时回写文件级敏感摘要。

\subsection{向量化与索引写入实现}

向量化阶段既要控制调用成本和批量吞吐，也要满足敏感信息不外传的合规要求。\texttt{VectorizationService} 通过 \texttt{EmbeddingModelClient} 调用外部嵌入模型，外送文本统一使用 \texttt{safeTextContent}；生成 2048 维向量后，再连同文件标识、权限信息、归一化文本和敏感元数据写入 \texttt{knowledge\_base} 索引。

\texttt{EmbeddingClient} 围绕重复请求消除、批量并行和并发回压组织调用控制。Caffeine 缓存与 \texttt{inFlightVectorCache} 分别用于复用已有向量和合并并发中的同文本请求；\texttt{embeddingBatchExecutor} 负责批量文本的并行处理；\texttt{AdaptiveEmbeddingConcurrencyLimiter} 则根据实时负载调整并发上限。

自适应并发限制器以 $\alpha=0.25$ 的指数加权移动平均（EWMA）跟踪时延趋势，使最近一次调用在限流决策中占更高权重。低时延阈值设为 1800~ms，高时延阈值设为 3500~ms：连续 $N_{\text{up}}=6$ 次健康完成后，并发上限增加 1，对应约 10~s 的稳定观测窗口，用于抑制并发上限在恢复阶段的频繁抖动；一旦单次时延或 EWMA 超过高阈值，或出现 429、5xx 响应，并发上限立即减 1，以便优先响应外部服务压力变化。

\section{混合检索与流式问答实现}
\label{sec:chat_impl}

问答链路由检索、生成与隐私治理共同构成。检索需要在单次查询内完成权限过滤与语义重排，生成需要维持多轮上下文边界，隐私治理则贯穿检索、外发与日志节点。以下各节依次展开。

\subsection{模型 Provider 抽象实现}

Provider 抽象的目标是隔离上层业务编排与厂商请求格式。系统将聊天模型与向量模型分别抽象为 \texttt{ChatModelClient} 和 \texttt{EmbeddingModelClient}，上层服务只依赖接口，不直接耦合具体厂商的请求结构。\texttt{AiProperties} 通过 \texttt{ai.providers.chat} 与 \texttt{ai.providers.embedding} 指定当前启用的 Provider；\texttt{AiModelProviderConfig} 则提供 \texttt{embeddingBatchExecutor} 线程池，以支撑向量批处理并行执行。

该边界约束将模型替换的影响范围收敛至 Provider 实现类与配置文件；接入新的对话模型或私有化向量服务时，会话调度、混合检索与异步入库逻辑均无需改动。

\subsection{混合检索实现}

混合检索把权限过滤、向量召回和词法重排收敛到同一请求中，同时将在线时延控制在可接受范围内。\texttt{HybridSearchService} 支持 \texttt{BM25\_ONLY}、\texttt{DENSE\_ONLY}、\texttt{HYBRID\_NO\_RESCORE} 和 \texttt{FULL\_HYBRID} 四种模式，系统默认使用 \texttt{FULL\_HYBRID}。检索入口先计算用户的有效组织标签，再执行对应的召回策略。

在 \texttt{FULL\_HYBRID} 模式下，单次 Elasticsearch 请求同时执行 kNN 向量召回与 BM25 文本匹配，两路结果在同一候选窗口内经 \texttt{rescore} 完成语义重排，取排名前 \textit{k} 条作为问答上下文（代码见下）。单次请求避免了两阶段检索的额外网络往返，权限过滤也可在 \texttt{filter} 子句中前置执行。对于多轮追问场景，\texttt{ChatHandler} 还可调用限定文档范围的检索接口，将候选集收缩到当前工作记忆中的活动文档集合。

\begin{lstlisting}[

 basicstyle=\ttfamily,

权限过滤在召回阶段前置执行，其作用域由“用户本人上传”“公开资源”与“用户有效组织标签范围”共同构成。该实现对应需求 R3，候选结果无需再在应用层进行事后裁剪。

系统通过 \texttt{ai.retrieval.*} 统一配置候选窗口、重排窗口和查询权重。对于短关键词查询，系统先执行 BM25 检索；若 Top-1 得分高于 \texttt{ai.retrieval.bm25-shortcut-score-threshold} 且与第二名的分差超过 \texttt{ai.retrieval.bm25-shortcut-gap-threshold}，则直接返回结果，跳过向量化步骤。对于数字密集型问题，\texttt{buildLexicalQuery} 会同时查询原始文本字段与 \texttt{normalizedTextContent} 字段，并提高归一化字段的匹配权重，使金额、比例和万/亿单位等表达在检索阶段保持稳定匹配。

检索链路出现异常时，系统按预设优先级依次降级：向量召回或重排不可用时退化为纯 BM25；文本检索亦不可用时直接返回无证据状态并终止本轮生成，避免在缺少证据支撑时继续输出。

\subsection{Prompt 工程实现}

Prompt 需要同时承载规则、历史与证据等异质输入。直接拼接为单一文本时，模型容易混淆指令层与上下文层。\texttt{ChatPromptAssembler} 因而将输入结构划分为若干具名层次：指令组包含回答规则、会话策略、记忆策略与隐私策略；上下文组包含轮次摘要与工作记忆摘要；证据组包含授权检索证据。各层的阈值、规则文案与占位文本由 \texttt{AiProperties} 集中管理，使 Prompt 结构在不同轮次和场景下保持稳定。

证据组为空时，装配器写入固定的占位文本，使有证据和无证据两种情形下的输入结构保持一致，减少模型行为波动。\texttt{DeepSeekClient} 只负责请求构造与流式回调，不承担 Prompt 规则维护职责。

\subsection{会话轮次路由与工作记忆实现}

多轮问答中，\texttt{ChatHandler} 在每轮开始时从 Redis 读取历史与工作记忆，再由 \texttt{ConversationTurnAnalyzer} 完成轮次分类。分析器输出四个判定维度：是否启动检索、是否继承历史上下文、是否限定文档范围，以及是否返回澄清提示，各维度的结果共同决定后续链路的执行路径。

\texttt{ConversationTurnAnalyzer} 会先识别显式新话题标记，命中后直接归为 \texttt{NEW\_TOPIC}；再识别 \texttt{META} 标记，把“回看历史”“总结前文”之类请求从检索链路中分离出来。若用户显式引用 \texttt{来源\#编号}，系统会把编号映射为文件 MD5，并优先进入限定文档的 \texttt{FOLLOW\_UP} 检索。

对于短追问，系统会结合短追问信号、上下文依赖型问题模式和活动作用域判断是否需要澄清。当输入看似追问，但工作记忆中缺少活动主题或活动文档时，链路直接返回澄清文案，不继续检索。其余追问场景则依据活动主题、活动主语、活动文档和关键词改写检索查询。

工作记忆由 \texttt{ConversationMemoryService} 维护，其中保存活动主题、活动主语、最近独立问题、最近答案摘要、活动文档和关键词等结构化信息。

历史注入策略分为 \texttt{SELECTIVE}、\texttt{RAW\_RECENT} 和 \texttt{DISABLED}。\texttt{SELECTIVE} 用于常规追问，从历史轮次中选取与当前问题相关的条目注入；\texttt{RAW\_RECENT} 用于长程回看，保留含有早期锚点的完整历史轮次；\texttt{DISABLED} 用于新话题切换，关闭历史继承。具体选择由 \texttt{ConversationTurnAnalyzer} 的分类结果自动决定。

\subsection{隐私治理实现}

同一文本在索引、外发与日志阶段面临不同的安全风险，隐私治理因此须沿链路分阶段介入。以下分别说明识别接口、分阶段保护策略与高风险拦截的实现。

\subsubsection{识别接口}
\texttt{SensitiveContentGovernanceService} 采用关键词字典匹配与正则模式识别相结合的方式，覆盖身份证号、手机号、电子邮箱、银行卡号、API Key 等个人信息及企业受限关键词。对每段文本，服务都会生成治理结构，记录脱敏文本、敏感级别、受限标记、外发许可标志与类型摘要，供索引、模型调用和日志处理复用。

\subsubsection{分阶段保护策略}
文档处理链路中，\texttt{ParseService} 在每个文本块入库前调用 \texttt{protectForIndexing}，同步写入安全文本、归一化文本、敏感级别与类型摘要。对外部模型调用前，系统使用 \texttt{protectForExternalModel} 处理用户问题、检索查询和候选证据；日志阶段只保留 \texttt{protectLogPreview} 生成的安全预览。三类视图的保护范围依次收紧：索引视图保留可检索文本并屏蔽高风险字段；外发视图在此基础上进一步校验 \texttt{externalModelAllowed} 标志，替换受限金额与比例等内容；日志视图则截断为固定长度的最小安全预览。

\subsubsection{高风险请求拦截}
在线问答链路中，Embedding 输入在外送前先完成治理；\texttt{ChatHandler} 在装配 \texttt{ChatPromptPayload} 时只使用处理后的问题、检索查询和历史文本；\texttt{buildContext} 会把检索证据包装为带 \texttt{sensitivity} 和 \texttt{external\_allowed} 属性的授权上下文片段；模型输出在流式返回前再执行一次结果保护。对于“导出原文”“忽略规则”等高风险请求，系统在模型调用前通过 \texttt{detectBlockedPromptReason} 直接拒答，使拦截逻辑位于链路入口。

\subsection{长连接流式交互实现}

流式交互的核心约束在于：增量内容推送、完成信号判定与引用来源映射须在同一链路中保持时序一致，任一环节的时序偏差都可能导致来源信息丢失或内容截断。前端聊天模块通过 WebSocket 长连接通信协议建立连接，并借助 \texttt{@vueuse/core} 提供的 \texttt{useWebSocket} 完成会话管理；连接建立后接收后端返回的 \texttt{sessionId}，后续消息以会话标识和用户问题作为基本载荷发送到服务端。后端在 \texttt{ChatWebSocketHandler} 中解析 JWT、恢复用户身份，再将文本消息交给 \texttt{ChatHandler}。

流式执行链路依赖会话级状态管理。\texttt{ChatHandler} 通过 \texttt{streamingTurnStates} 保存当前轮次响应状态，以 \texttt{stopFlags} 管理中断标志，以 \texttt{sessionReferenceMappings} 维护引用编号到文件 MD5 的映射；完成超时与停止标志清理则由 \texttt{completionWatcherExecutor} 统一调度，相应参数受 \texttt{ai.streaming.*} 配置约束。流式返回时，模型增量内容被序列化为 \texttt{\{"chunk": "..."\}} 形式的 JSON 消息并逐段推送；当 \texttt{onComplete} 回调触发，或超时 watcher 判定需要强制收尾时，服务端再发送单独的完成通知。回答文本、引用来源与编号映射沿同一路径写入。

会话记录在收到完成信号后一次性落盘，并同步回写工作记忆，以降低高频写入带来的存储压力。客户端若在生成过程中断连，服务端检测到写入异常后立即停止推送并释放连接状态；只有回答文本与来源映射同时完整时，本轮记录才会落盘。


本章涉及的四类对象分别服务于解析、向量化、检索与文件状态管理，但其字段命名保持对应关系。例如，\texttt{safeTextContent} 与 \texttt{sensitivityLevel} 从文本块阶段延续到索引与接口层，使安全视图和风险标记在各阶段共享统一的字段语义。基于这组字段设计，文档列表接口可直接返回文件名、组织标签名称、敏感级别、受限状态和敏感类型等信息；会话历史接口也沿用同一套语义组织返回结果。
