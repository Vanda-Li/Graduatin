# 论文答辩 PPT 重设计方案（科研看板风，8 分钟版）

适用论文题目：基于 RAG 的企业私有文档智能问答系统

本文档不再沿用普通“项目汇报型”答辩 PPT 的写法，而是参照你给出的那类“评委爱看”的科研答辩页面风格，重新组织整套答辩内容。核心目标不是把论文逐章缩写，而是把每一页都做成“结论明确、证据集中、模块整齐、扫一眼就知道你要说什么”的科研看板。

这套方案特别适合你的论文。因为你的论文本身不是单点算法，而是“问题定义清楚、系统边界明确、关键机制可拆开讲、实验结果可用数字支撑”的系统设计型论文，天然适合这种蓝白科研答辩风格。

## 一、参考图风格分析

你给的三张参考图，本质上不是“设计感 PPT”，而是“科研信息看板式 PPT”。它们之所以比普通学生自制 PPT 更受评委欢迎，原因主要有六点。

### 1. 不是“随便摆文字”，而是“有层次的信息板”

参考图的每一页都有非常稳定的层次：

- 左上是章节编号，例如 `1.1`、`1.2`、`2.1`。
- 顶部蓝条是页面主标题。
- 蓝条下面还有一条“结论条”，直接告诉老师这一页要证明什么。
- 中部是分块信息区，每块都有边框、有标题、有配图或示意。
- 底部再用一句话把本页结论收住。

这意味着老师不是在“猜这页想说什么”，而是在“按你设计好的顺序读取这页”。

### 2. 每页只有一个核心问题，不做散点堆砌

参考图虽然信息量不小，但每页都围绕单一主题展开：

- 一页只讲立项意义。
- 一页只讲科学问题。
- 一页只讲研究思路。

普通学生 PPT 常见的问题，是一页里同时出现背景、方法、实验、结论和一堆配图，导致看起来很满，但没有重心。参考图的优点恰恰是“信息多但不散”。

### 3. 配图不是装饰，而是证据

参考图里的飞机、直升机、结构示意、实验照片，不是为了“让页面好看”，而是为了给结论提供支撑。也就是说，图像在这里承担的是论证功能，而不是壁纸功能。

这点对你的论文尤其重要。你的答辩 PPT 不能靠“AI 科技背景图”来撑场面，而要靠：

- 系统架构图
- 文档链路图
- 权限判定图
- 会话控制图
- 隐私治理图
- UI 页面截图
- 实验指标卡

这些图才是你的“证据图”。

### 4. 颜色服务于信息层级，而不是追求花哨

参考图的颜色使用非常克制，基本规律是：

- 深蓝：标题条、章节条、主结构
- 浅蓝：模块底板、信息卡
- 红色：核心问题、关键痛点、结论强调
- 黑色：正文
- 白色：背景

这类配色非常适合工科和系统论文，因为它会让页面看起来像“学术汇报材料”，而不是“互联网产品发布会”。

### 5. 页面像一张缩微海报，适合答辩场景

参考图的每一页都像一张“小海报”，非常适合答辩。因为评委看 PPT 的方式不是逐字阅读，而是：

1. 先看标题和结论条；
2. 再扫中间几个模块；
3. 最后听你口头把它们串起来。

所以，真正适合答辩的 PPT，不是“每个字都写得完整”，而是“视觉顺序清楚，讲述时容易带着老师走”。

### 6. “自己瞎做的”那种页面，问题不在丑，而在信息组织失败

你给的对比图里，上面那种“自己瞎做的”页面主要有这些问题：

- 没有结论条，老师看不出这一页到底要证明什么。
- 文字与图片是松散摆放，没有模块边界。
- 所有信息权重差不多，没有主次。
- 图片太小且无组织，既不能形成冲击，也不能承担论证功能。
- 页面只是“把内容放上去”，而不是“把论证关系设计出来”。

因此，你这次答辩 PPT 的重点，不是单纯把页面做精致，而是把“结论-模块-证据-收束”这套结构真正搭起来。

## 二、将这种风格翻译到软件系统论文中的原则

你论文是软件系统方向，不是材料、机械或航空方向，所以不能机械照搬参考图中的图片题材，但要保留它的信息组织方式。

### 1. 要借鉴的是“结构”，不是“题材”

参考图里的飞机、试样、热防护照片，对应到你的论文里，应替换为：

- 企业文档与知识服务场景图
- 系统架构图
- 核心流程图
- 安全链路图
- UI 界面图
- 实验结果图

不要去模仿材料类论文的工业照片感，而要把软件论文自己的“结构图、流程图、结果图”用科研看板方式摆出来。

### 2. 你的 PPT 应该是“图主文辅”

建议遵循下面的比重：

- 60%：图、框、流程、模块、指标卡
- 25%：短句说明
- 15%：口头补充

不要让一页出现超过 120 字的正文。能用图表达的，不用段落。

### 3. 外部网图只做辅助，主视觉必须来自论文本身

对你的答辩来说，最重要的图像来源不是网上配图，而是你自己的论文图和系统截图。外部网图只适合放在：

- 封面淡化背景
- 研究背景页
- 研究问题页的小图标或示意插图

- 全屏蓝紫色赛博背景图
- 发光机器人头像
- 随机代码屏幕截图
- 与论文无关的芯片、地球、光束、城市夜景
- 过于夸张的 3D 数据大屏素材

这些图看起来“像 AI”，但并不能帮助老师理解你的论文。

## 三、统一模板规范

为了让整套 PPT 真正像参考图那样统一，建议先做母版，再填内容。

### 1. 颜色

推荐配色如下：

- 主蓝：`#1F4E9E`
- 深蓝：`#163B78`
- 浅蓝底：`#F3F7FD`
- 分隔线蓝灰：`#AEBED8`
- 强调红：`#C73C32`
- 强调黄：`#F2B236`
- 正文黑：`#222222`

建议规则：

- 蓝色负责结构。
- 红色负责问题与结论强调。
- 黄色只用于顶部一句话结论中的关键词，不要大面积使用。

### 2. 字体

建议如下：

- 中文标题：思源黑体 Heavy / 微软雅黑 Bold
- 中文正文：思源黑体 / 微软雅黑
- 英文与数字：Times New Roman 或 Aptos

字号建议：

- 页面主标题：26 到 30 pt
- 结论条：18 到 22 pt
- 模块标题：18 到 20 pt
- 正文：14 到 18 pt
- 指标数字：24 到 32 pt

### 3. 页面骨架

每一页尽量遵循同一骨架：

```text
┌ 编号条 + 页面标题 ────────────────────────────────┐
├ 顶部结论条：一句判断句 ───────────────────────────┤
│                                                   │
│  中部 2~4 个模块：流程 / 图 / 指标 / 对比 / 截图   │
│                                                   │
├ 底部收束句：一句归纳或一句过渡 ───────────────────┤
└───────────────────────────────────────────────────┘
```

### 4. 模块样式

建议所有中部模块统一为：

- 浅蓝或白色底
- 细灰蓝边框
- 顶部小标题条
- 内部图像尽量充满模块，不留太多白边
- 说明文字控制在 3 条以内

### 5. 动画

只建议做两类动画：

- 模块依次出现
- 指标卡依次出现

不要使用飞入、翻转、旋转、弹跳等强动效。

## 四、整套 PPT 新结构

建议采用“封面 1 页 + 目录 1 页 + 主讲 8 页 + 致谢 1 页 + 备份 3 页”的结构。

### 主讲页结构

| 页码 | 页面编号 | 页面主题 | 建议时长 |
| --- | --- | --- | --- |
| 1 | 封面 | 题目、作者信息、研究定位 | 10 秒 |
| 2 | 目录 | 四段式目录 | 15 秒 |
| 3 | 1.1 | 研究背景 | 45 秒 |
| 4 | 1.2 | 核心问题 | 45 秒 |
| 5 | 2.1 | 研究思路与主要贡献 | 55 秒 |
| 6 | 3.1 | 系统总体架构 | 55 秒 |
| 7 | 3.2 | 关键设计一：可靠接入与权限前置 | 65 秒 |
| 8 | 3.3 | 关键设计二：会话状态与隐私治理 | 70 秒 |

### 第 3 页：1.1 研究背景

**页面核心结论**

企业私有文档问答的难点，不只是“模型能不能回答”，而是“系统能不能在企业约束下稳定运行”。

**顶部结论条文案**

`现有企业知识问答不仅要求回答有效，更要求在持续接入、权限约束和多轮交互条件下保持可信`

**推荐布局**

```text
┌ 左侧：两张背景图 ───────────────┬ 右侧：三条压力 ──────────────┐
│ 企业文档/办公场景               │ 1. 文档持续更新                │
│ 企业协作/知识服务场景           │ 2. 多组织权限并存              │
│                                 │ 3. 多轮交互与安全风险          │
└─────────────────────────────────┴──────────────────────────────┘
底部：一句归纳
```

**推荐本地图**

- 无需放复杂论文图

**可选在线配图**

- `Pexels` 文档办公场景图
- `Pexels` 团队协作或笔记本办公图

**底部收束句**

`因此，本文研究的不是普通聊天原型，而是面向真实企业文档流转过程的问答系统`

**话术**

企业私有文档问答和开放域聊天不同，它不仅要求模型能生成回答，还要求系统能在真实业务约束下长期稳定运行。企业文档会持续更新，权限边界会随组织关系变化，多轮追问又会引入证据漂移和敏感信息外泄风险。因此，本文的问题定义从一开始就不是“做一个聊天页面”，而是构建一套可持续运行、可授权、可追溯、可治理的问答系统。

---

### 第 4 页：1.2 核心问题

**页面核心结论**

现有 RAG 研究普遍默认“静态语料 + 后置权限裁剪”，难以直接适配企业私有文档场景。

**顶部结论条文案**

`主流 RAG 以静态语料为前提，尚未直接回答企业文档持续接入、权限前置和多轮安全协同问题`

**推荐布局**

```text
三列问题卡
[问题一] 文档接入链路长、在线阻塞
[问题二] 权限过滤后置、边界不一致
[问题三] 多轮追问易漂移、敏感信息易扩散
```

每一列建议结构：

- 上：问题标签
- 中：一句具体说明
- 下：一个小图标或小示意图

**可选在线配图**

- `unDraw` 或 `Storyset` 的文件、认证、聊天、安全类插图各一张

**底部收束句**

`因此需要把“接入、检索、会话、安全”放在同一系统边界内统一设计`

**话术**

论文分析表明，现有 RAG 研究大多默认知识源已经完成清洗和索引，权限控制也往往在召回之后再做后置裁剪。这种前提在企业私有文档场景里并不成立。对企业来说，真正的问题至少包括三类：第一，长文档上传后的解析和向量化会阻塞在线链路；第二，权限边界如果不进入候选集生成阶段，就无法保证授权一致；第三，多轮追问会导致证据边界漂移，并增加敏感信息扩散风险。

### 第 6 页：3.1 系统总体架构

**页面核心结论**

### 第 10 页：4.2 实验结果与结论

**页面核心结论**

实验结果说明，系统在检索效果、响应性能和安全边界三方面均达到预期目标。

**顶部结论条文案**

`实验最终回答了三个问题：能不能检到、能不能及时响应、能不能守住安全边界`

**推荐布局**

这一页不要做成表格朗读页，建议做成“三问三答”的答辩页。

```text
┌ 检得准 ┐ ┌ 响应快 ┐ ┌ 守得住 ┐
│ Recall@5 │ │ P95/TTFT │ │ 越权/泄漏 │
│ 多轮对比  │ │ 并发/QPS │ │ 对抗提示   │
└────────┘ └────────┘ └────────┘
底部：一条总判断
```

**三块建议数据**

1. 检得准
   - 正式配置 `Recall@5 = 0.9800`
   - `MRR@5 = 0.9051`
   - `nDCG@5 = 0.8562`
   - 选择性历史语义通过率 `0.7778`
   - 对比：无历史 `0.4444`，近轮拼接 `0.6889`

2. 响应快
   - 检索响应 `P95 = 1099.0 ms`
   - `TTFT P95 = 1701.0 ms`
   - 峰值 `QPS = 650.83`
   - 79.0% 文档在 2 秒内完成可检索就绪

3. 守得住
   - 30 组权限矩阵零越权
   - 45 条对抗提示全部通过
   - 当前测试范围内泄漏率为 `0`

**特别说明**

如果答辩时需要补充“为什么混合检索有效”，放到备份页，不要把这一页变成消融表朗读页。主讲页只回答三件事：

- 效果是否达到目标
- 性能是否满足系统约束
- 安全边界是否可靠

**底部收束句**

`结果表明，系统能够在企业私有文档场景下兼顾问答有效性、授权一致性与内容安全`

**话术**

实验结果我重点回答三个问题。第一，系统能不能检到正确证据。正式配置下，Recall@5 达到 0.98，多轮场景中选择性历史策略也明显优于无历史和近轮拼接基线。第二，系统响应能不能接受。在线检索响应 P95 为 1099 毫秒，首字等待时间 P95 为 1701 毫秒，1000 用户并发下峰值 QPS 为 650.83。第三，安全边界能不能守住。在当前测试范围内，30 组权限矩阵零越权，45 条对抗提示全部通过，隐私治理链路下泄漏率为 0。综合来看，系统在效果、性能和安全三方面都达到了论文预期。

### 第 11 页：5.1 总结与展望

**页面核心结论**

本文验证了企业私有文档问答系统在统一架构下实现可靠接入、授权一致和内容安全的工程可行性。

**顶部结论条文案**

`本文的价值不在于单点模型优化，而在于把接入、检索、会话与治理统一为一套可运行系统`

**推荐布局**

```text

```

**左侧三条结论**

1. 构建了统一架构下的企业私有文档问答系统
2. 验证了权限前置、会话控制和隐私治理的工程有效性
3. 在效果、性能和安全三方面达成既定目标

**右侧局限与展望**

局限：

- 检索与生成仍主要基于单模型组合，模型侧比较不充分
- 困难场景下的交叉比较与模糊追问澄清能力仍有提升空间

展望：

- 引入更强的比较型与澄清型对话策略
- 完善多模型切换、可观测性和更大规模测试

**底部收束句**

`后续工作将重点从“系统可行”推进到“系统更强、更稳、更可观测”`

**话术**

最后做一个总结。本文的主要贡献不是提出一个新的大模型算法，而是在企业私有文档场景下，把文档接入、授权检索、多轮会话和隐私治理放到同一系统边界内统一设计，并通过实验验证其工程可行性。当前系统已经证明了这一整体方案是可运行、可验证的，后续工作将进一步加强复杂多轮场景、模型切换与系统可观测性。

## 八、在线图片的使用规范

这部分很重要。为了做出“像参考图那样”的效果，网图一定不能乱用。

### 1. 图片只承担这三类角色

- 场景引入
- 模块点题
- 氛围补充

不能承担这两类角色：

- 核心机制说明
- 实验结论证明

后两类必须由你的论文图和指标图承担。

### 2. 一页最多放 2 张外部图片

超过 2 张，页面会开始像海报素材拼贴，而不是科研汇报。

### 3. 同一页尽量统一风格

不要一边用真实照片，一边用 3D 插画，一边用扁平图标。建议一页只用一种外部图风格：

- 要么都是照片
- 要么都是线稿插图
- 要么都是同一套扁平插图

### 4. 图片颜色要主动调成蓝白体系

对 `unDraw` 和 `Storyset` 插图，建议全部调成主蓝 `#1F4E9E` 附近，这样能和整套 PPT 统一。

### 5. 不要让图片比结论更抢眼

评委最先看到的应该是：

- 页面标题
- 顶部结论条
- 中间主模块

而不是“这个图挺好看”。

## 十、制作时必须遵守的几条原则

- 不要把论文原图整张缩小后直接塞进 PPT。
- 不要在一页里堆超过 4 个并列重点。
- 不要把网图当主视觉，把自己的论文图挤到角落。
- 不要把实验页做成表格朗读页。
- 不要把系统实现页做成“截图拼盘”，每个截图必须有标题。
- 不要使用全屏深色背景；这套风格最适合白底蓝条。
- 不要把“系统设计”“实验结果”这种中性词当标题，尽量把标题写成判断句。

推荐标题写法：

- 不推荐：`系统设计`
- 推荐：`系统通过分层架构实现接入、检索与模型解耦`

- 不推荐：`实验结果`
- 推荐：`实验结果说明系统在效果、性能和安全三方面达到预期`
