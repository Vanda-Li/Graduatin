# 第五章全面重跑测试A计划书（审阅版）

更新时间：2026-04-05

## 1. 计划目标

本计划用于解决第五章当前“新旧测试结果混用、统计口径不统一、章节叙述前后不一致”的问题，目标是在不再引用旧跑次结果的前提下，重新启动一轮可直接回填论文第五章的全面新实验。

本轮计划采用以下统一口径：

- 文档总量固定为 `200` 份。
- 全部 `200` 份文档均执行完整入库。
- 全部 `200` 份文档均参与检索评测，不再以 `formal-50` 作为第五章主检索口径。
- 并发性能采用梯度用户并发，档位为 `50 / 100 / 200 / 500 / 1000` 用户。
- 第五章正文只引用同一轮重跑生成的新结果，不再混用旧的 `formal-50`、`pressure-200`、`u50`、`u150` 或复用基线结果。

## 2. 当前问题归纳

结合 [chap05.tex](/home/lilu/Graduatin/docs/chap05.tex) 与 Ark 当前实验实现，现阶段至少存在以下混口径问题：

- 第五章入库部分已经按 `200` 份正式样本书写，但检索主结果仍以 `50` 份 `formal-50` 数据集为主。
- 当前 `200` 份检索结果实际存在一套 `pressure-200` 结果，但其 `200` 条查询全部为 `EXACT_MATCH`，更适合做压力复核，不适合作为论文主检索效果证明。
- 当前性能段落的并发结果仍写成单一 `50` 用户并发，既不能体现系统扩展性，也与本轮梯度并发目标不一致。
- 代码中的 chapter05 专项测试仍残留“复用旧稳定基线”和“旧大样本/旧并发档位”的设计，例如 [Chapter05ReviewSystemExperimentTest.java](/home/lilu/Ark/src/test/java/com/yizhaoqi/ark/experiment/system/suite/chapter05/Chapter05ReviewSystemExperimentTest.java) 与 [SystemConcurrentProfileRunner.java](/home/lilu/Ark/src/test/java/com/yizhaoqi/ark/experiment/system/module/retrieval/perf/SystemConcurrentProfileRunner.java)。
- 当前 focused 结果目录中同时存在 `formal-50`、`holdout-50`、`medium-50`、`pressure-200`、`50 用户并发阶梯` 等多套结果，若不先清理，后续极易再次混入章节引用。

### 3.1 样本规模硬约束

本轮新增一条统一约束：

- 所有正式测试项的测试数据至少达到两位数数量级。
- 关键测试项应优先达到三位数数量级。
- 单位按测试对象确定，可以是“文档数”“查询数”“检查组数”“会话数”“轮次数”“提示数”或“请求数”，但不得出现个位数主样本。

本轮计划的最低规模要求如下：

| 测试项 | 最低规模要求 | 本轮计划值 |
| --- | --- | --- |
| 全链路入库 | 文档数 `>= 100` | `200` 份文档 |
| 大文档续传 | 探针数 `>= 10` | `20` 份 |
| 边界条件 | 检查项 `>= 10` | `15` 项 |
| 主检索评测 | 文档数 `>= 100`，查询数 `>= 300` | `200` 份文档，`400` 条查询 |
| 权限矩阵 | 检查组数 `>= 20` | `30` 组 |
| 流式交互 | 会话数 `>= 10`，轮次数 `>= 30` | `10` 个会话，`30` 轮 |
| 多轮语义与历史 | 场景数 `>= 10`，轮次数 `>= 50` | `10` 组场景，`60` 轮 |
| 文件服务 | 检查项 `>= 20` | `40` 项 |
| 敏感信息与防泄漏 | 样本或提示数 `>= 10` | `20` 份 |
| 检索模式对照 | 查询数 `>= 300` | 复用 `400` 条主查询 |
| 历史策略对照 | 场景数 `>= 10`，轮次数 `>= 50` | 复用 `10` 组场景，`60` 轮 |
| 并发性能 | 至少 `5` 个并发档位，最高档用户数 `>= 1000` | `50 / 100 / 200 / 500 / 1000` 用户，合计 `3700` 请求 |

执行约束：

- 若某项测试样本规模低于上表计划值，则该项结果不得进入第五章正式结论。
- 若后续因成本原因缩减样本，缩减后仍不得低于“两位数”下限。
- 第五章中最关键的主结果优先保持三位数规模，包括入库文档数、主检索查询数、并发高档用户数或请求数。

### A3. 200 份全量检索主评测

目的：用同一批 `200` 份正式文档建立第五章新的主检索结论。

核心原则：

- 第五章主检索结果不再使用 `formal-50`。
- 第五章主检索结果也不直接使用当前 `pressure-200`，因为该数据集 `200` 条查询全部为 `EXACT_MATCH`，不足以证明真实混合检索能力。
- 本轮需新建“full-200 正式检索 benchmark”。

建议查询集规模：

- 推荐主查询集规模为 `400` 条。
- 推荐按“全量文档覆盖 + 查询类型均衡”设计。
- 推荐每份文档至少覆盖 `1` 条主查询，重点难例文档补充 `2` 至 `3` 条。

建议查询类型分布：

| 查询类型 | 建议数量 |
| --- | ---: |
| EXACT_MATCH | 40 |
| ENTITY_ABBREVIATION | 40 |
| SEMANTIC_PARAPHRASE | 60 |
| TOPIC_GENERALIZATION | 60 |
| QUESTION_QUERY | 40 |
| ATTRIBUTE_DESCRIPTION | 30 |
| TEMPORAL_CONSTRAINT | 30 |
| CONTENT_DETAIL | 20 |
| COMPARATIVE | 20 |
| INDUSTRY_DOMAIN | 20 |
| MULTI_CONSTRAINT | 20 |
| VAGUE_MEMORY | 20 |

查询设计要求：

- 必须覆盖全部 `8` 类内容类型。
- 必须覆盖 PDF、DOC/DOCX、XLS/XLSX、TXT 四类文件格式。
- 必须覆盖相似标题、年份差异、修订版/更正版/更新版干扰。
- 必须覆盖简称、别名、语义改写、属性询问、时间约束和多约束组合。
- 必须为容易误召回的附件类文档提供单独难例。
- 每条查询都记录目标文档、可接受结果、混淆文档和权限属性。

标注字段与判定标准：

- 每条查询至少记录 `queryId`、`queryText`、`queryType`、`targetDocumentIds`、`acceptableDocumentIds`、`confusableDocumentIds`、`permissionAttributes`、`contentCategory`、`fileType`、`evidenceNote`、`ambiguityNote`、`annotator`、`reviewer` 和 `reviewStatus`。
- 单目标查询必须满足“主要目标文档唯一且可通过标题、公司名、年份、主题或属性约束稳定指向”；若一个查询天然能命中多份文档，则不得强行按单目标记分。
- 只有当查询在复核后可以稳定判定“哪些结果应算命中、哪些结果仅为干扰项”时，才允许进入正式 benchmark；依赖隐藏元数据、OCR 偶然噪声或人工脑补上下文的查询不得入库。
- 多目标查询必须单独标注全部可接受结果，并在统计时纳入 `multi-expected` 子集，不与单目标查询混算首位命中质量。
- benchmark 冻结后不得临时改写查询文本或放宽可接受结果集合；若确需修改，必须增加版本号并保留变更记录。

歧义处理规则：

- 若歧义可以通过补充公司名、年份、附件类型、报告阶段或主题词消除，则先改写查询，再进入正式集。
- 若歧义来自真实业务语义，且多个结果都合理，则将其改标为多目标查询，并完整记录可接受结果集，不再把它当作“唯一标准答案”。
- 若同一查询经试标和复核后仍无法稳定达成一致，则移出正式 benchmark，进入“争议样本池”，不得用于第五章正式主结果。
- 所有争议条目都要保留仲裁记录，说明争议原因、处理结论与最终版本。

标注组织与可重复性要求：

- 初标由论文作者完成，复核由第二标注者完成；在正式批量标注前，先做 `20` 条试标并统一规则。
- 正式阶段不少于 `20%` 的查询做双人独立复标；若复标结论不一致，必须回到规则表裁决并统一修订。
- 标注模板、字段说明、版本号和变更记录必须随 benchmark 一起保存，保证后续重跑可以复用同一口径。
- 若某一类文档在两轮设计后仍无法形成稳定高质量查询，不允许为了凑数量硬塞弱查询；应降低该类目标占比、记录原因，并在计划书和最终报告中明示调整。

Excel/TXT 附件查询的特殊处理：

- Excel 附件查询优先围绕表头、字段名、行列关系、期间约束和附件标题设计，不以截图视觉结构、复杂合并单元格版式或图表观感作为唯一判定依据。
- TXT 附件查询优先围绕附件标题、主题关键词、短事实片段和上下文限定设计，避免只用极短通用词组导致偶然命中。
- Excel 和 TXT 附件查询在文本本身区分度不足时，必须显式加入“附件/表格/清单/说明”之类附件标识，或补充所属公司、年份和主题词。
- 若某份 Excel 或 TXT 附件无法稳定支撑至少 `1` 条高质量查询，则该文档不强制纳入主 benchmark，可转入补充样本并在报告中说明原因。

采集指标：

- Recall@1、Recall@5。
- MRR@5。
- nDCG@5。
- 平均时延、P50、P95。
- 各查询类型分组结果。
- 典型失败样例。

建议门槛与失败处理：

- 时延硬门槛仍按第五章非功能需求阈值控制 `P95 <= 1500 ms`。
- 主检索质量硬门槛设为 `Recall@5 >= 0.95`；低于该值时，结果不得直接进入第五章正式主结论。
- 若本轮 `Recall@5` 落在 `0.95` 到旧 `formal-50` 结果 `0.98` 之间，可作为新的 full-200 正式结果接受，但正文必须明确说明“数据集扩大、查询更难、附件比例提高”带来的差异，不能简单宣称性能全面提升。
- 若本轮 `Recall@5 < 0.95`，或核心类型分组中出现明显失衡，则必须先做失败分析，再决定是否进行一次针对性修复与重跑；在原因未查清前，不回填第五章主结果。
- 失败分析至少输出“按查询类型、内容类型、文件类型和附件类别分组的失分原因”，用于区分“系统真实短板”与“标注/数据设计问题”。

正文回填位置：

- “检索与问答验证”主段。
- “检索与问答需求验证结果”表。
- “代表性查询类型下的检索结果”表。
- “功能模块需求验证结果汇总”中的检索条目。
- “主要性能指标与对比基线”中的检索响应时延行。

### A4. 权限、流式问答、历史承接与文件服务重跑

目的：保证第五章功能验证与安全验证全部来自同一轮新实验，而不是拼接旧补测结果。

样本规模下限与计划值：

- 权限矩阵：计划 `30` 组。
- 流式交互：计划 `10` 个会话、`30` 轮。
- 多轮语义场景：计划 `10` 组、`60` 轮。
- 文件服务检查：计划 `40` 项。
- 敏感信息问答样本：计划 `20` 份。

规模合理性说明：

- 流式交互的主要风险集中在建连、首字等待、完成通知、引用标记和连续追问承接，而不是依赖数百个独立样本估计总体均值；因此 `10` 个会话、`30` 轮的目标被视为第五章功能性验证的最低可接受正式规模。
- 多轮历史场景采用的是人工设计的依赖链，而不是独立同分布查询；`10` 组、`60` 轮可以覆盖省略指代、上一轮回指、主题切换、长程回溯、跨文档干扰和历史分页读取等核心风险。
- 若流式或历史测试在某一类场景上出现失败，不应只保留原始 `10/30` 或 `10/60` 规模直接收尾，而应对失败类别补充至少 `2` 个会话或 `1` 组场景做回归确认。

执行原则：

- 权限探针采用本轮运行时动态上传的受控文档，不复用旧样本。
- 流式问答和历史承接所引用的文档，必须来自本轮已完成入库的 `200` 份正式集。

### A6. 梯度用户并发正式实验

目的：刷新第五章在线性能与并发结论，并通过多档位并发结果体现系统扩展性与负载增长趋势。

统一口径：

- 采用梯度用户并发，档位固定为 `50 / 100 / 200 / 500 / 1000` 用户。
- 建议每用户请求数统一为 `2`。
- 低档和中档用于观察系统稳定能力，高档用于观察扩展性边界与退化趋势。

建议并发档位：

| 档位 | 用户数 | 每用户请求数 | 总请求数 | 定位 |
| --- | ---: | ---: | ---: | --- |
| C1 | 50 | 2 | 100 | 低档稳定性基线，与旧第五章口径保持可比性 |
| C2 | 100 | 2 | 200 | 中档正式能力基线 |
| C3 | 200 | 2 | 400 | 高负载稳定能力验证 |
| C4 | 500 | 2 | 1000 | 扩展性压力观察 |
| C5 | 1000 | 2 | 2000 | 扩展上限观察 |
| 合计 | 1850 | - | 3700 | 梯度并发总观测量 |

并发查询设计：

- 不再沿用“单探针热查询”作为第五章唯一并发口径。
- 建议从 `full-200` 检索 benchmark 中选取不少于 `100` 条代表性查询组成混合并发池。
- 查询池需覆盖精确匹配、简称、语义改写、内容细节、附件检索五类场景。
- 查询池需覆盖全部 `4` 类文件格式，并尽量覆盖 `8` 类内容类型中的多数类别，最低不少于 `6` 类。
- 同一虚拟用户的两次请求不得落到同一条查询；前 `100` 次正式请求原则上不应出现重复查询。
- 用户请求按轮转或随机方式分配到查询池，避免结果只反映单一缓存热命中情况。

缓存控制策略：

- 正式并发报告必须显式标注本次结果属于“冷应用/热索引”还是“热应用/热索引”，不得只给出单个 P95 数字而不说明缓存状态。
- 正式主口径建议采用“冷应用、热索引、禁用应用层检索结果缓存”方案：启动前重启应用或清空应用进程内缓存，清理 Redis 中与本轮测试相关的会话和检索临时键，但不强求清空操作系统页缓存和 Elasticsearch 底层页缓存。
- 若系统实现中存在检索结果缓存、问答结果缓存或同类应用层复用机制，正式并发实验必须关闭；若代码中不存在该类缓存，也必须在报告中明确写出“应用层检索结果缓存不存在”。
- 允许保留认证、连接池、JIT 预热等运行所必需的基础热身，但这些 warmup 请求不得计入正式统计；同时不得使用正式查询池本身做预热。
- 若条件允许，建议额外补一组“热应用”对照结果，作为附录或补充说明，用于解释缓存命中对吞吐和 P95 的影响，但第五章主结果只保留一种明确口径。

采集指标：

- 各档位用户数、请求数与成功率。
- 成功率。
- 平均时延、P95、P99。
- QPS。
- 错误数与错误类型。
- Hit@5 命中率。
- 并发期间 CPU、内存、Kafka backlog、ES 查询耗时快照。

结果表达要求：

- 第五章中至少给出一张“梯度用户并发结果表”，展示 C1 到 C5 五个档位的成功率、Avg、P95、P99、QPS 和错误率。
- 第五章中至少给出一张“用户数-时延/QPS 变化曲线”，体现从 `50` 到 `1000` 用户时系统的扩展趋势。
- 正文必须区分“稳定能力档位”和“扩展性观察档位”，不能把 `500` 或 `1000` 用户结果与 `50` 或 `100` 用户结果混写成单一结论。

建议门槛：

- C1 至 C3 作为稳定能力档位，建议成功率 `>= 99%`，P95 检索时延 `<= 1500 ms`。
- C4 至 C5 作为扩展性观察档位，重点记录退化趋势、错误率和 backlog 回收情况，不强行套用与低档相同的硬时延门槛。
- QPS 作为记录指标，不单独设死门槛。
- 正式结果必须附带缓存控制说明、warmup 处理方式、查询池大小与查询去重策略，否则该轮并发结果不进入第五章正式主结论。

正文回填位置：

- “性能验证”段落。
- 梯度用户并发表。
- 并发时延分位图或用户数-时延增长曲线。
- “主要性能指标与对比基线”中的并发行。
- “非功能需求验证结果汇总”中的 NF1 行。
