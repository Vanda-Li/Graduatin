# 系统实现类工程论文写作结构化规约

## 1. 文档目的

本规约面向系统设计、系统实现、平台开发、信息系统、工程落地类本科毕业论文写作，目标不是提供套话模板，而是建立一套可执行的写作约定，使全文在结构、段落、句法、措辞和论证方式上保持学术性、工程性与可读性。特别地，本规约旨在**彻底消除大语言模型（AI）生成的“八股文”痕迹、机器味废话和空心化表达**。

本规约默认服务于如下类型的论文：

- 以一个具体系统、平台或工程项目为研究对象
- 以需求分析、系统设计、系统实现、系统测试为主线
- 强调工程闭环，而非纯理论推演
- **必须避免明显的 AI 套话、结构模板化、技术百科全书式默写和语言空转**

---

## 2. 总体写作原则

### 2.1 论文主线

全文必须围绕一条稳定的工程论证链展开：

`问题界定 -> 需求提出 -> 方案组织 -> 关键实现 -> 测试验证 -> 不足与后续方向`

任何章节的写作都不得脱离这一主线。若某一节无法说明自己在这条链上的位置，则该节应删改或并入相邻部分。

### 2.2 核心约定

1. 论文写的是“问题如何被界定、组织、落地和验证”，不是“开发过程回放”，更不是“技术组件的说明书”。
2. 章节之间必须前后闭合。需求中提出的约束，要在设计中得到回应；设计中提出的机制，要在实现中得到落地；测试中呈现的结果，要回到需求项上作判断。
3. 语言以准确、节制、客观、清晰为准，不以热闹、饱满、华丽为目标。
4. 评价性表述必须可回到需求、设计或测试结果（用数据或事实说话），不允许空泛拔高。
5. 就需求论测试，就结果论需求，不写提示性、过程性、教程式描述。

### 2.3 明确禁则（高危词汇黑名单）

全文原则上绝对禁止出现下列表述方式：

**1. 过程与代码级废话：**
- 根据代码仓库……
- 根据测试程序……
- 从源码可以看出……
- 如下图所示点击……
- 在这里可以看到……
- 先进入……再点击……

**2. 假大空的赞美词（AI最爱）：**
- 大大提高……
- 极大增强……
- 十分方便……
- 功能强大……
- 界面美观……
- 具有重要的理论意义和现实意义

**3. 强行总结与强加因果（典型AI逻辑缝合剂）：**
- 总而言之 / 综上所述 / 简而言之
- 这意味着 / 换而言之 / 由此可见 / 这表明了
- 需要指出的是 / 值得注意的是 / 显而易见
- 为后文奠定基础 / 为下一章开展奠定坚实基础

上列短语在工程论文中往往意味着论证失效、表述发空或机器生成痕迹过重，应作为**默认无条件删改对象**。

## 3. 全文结构约定

### 3.1 推荐结构

系统实现类工程论文优先采用六章或七章结构。

**六章型**
1. 绪论
2. 相关技术或需求准备
3. 系统需求分析
4. 系统设计
5. 系统实现与系统测试
6. 总结与展望

**结构选择规则**
- 技术路线对后文约束明显时，单列“相关技术”一章。
- 测试内容完整、结果具有分析价值时，单列“系统测试”一章。
- 系统规模较小、测试内容有限时，可将实现与测试合并，但测试小节仍须独立出现。

### 3.2 严禁流水账式的“本章小结”

- **废除八股文小结：** 严禁使用“本章首先介绍了……其次分析了……最后实现了……为下一章的开展奠定了坚实的基础”这种复读机式的废话。
- **强制要求：** 如果必须有小结，不许使用行为动词，必须直接陈述本章得出的**客观结论**或**交付的工程产物**。如果某章篇幅较短、逻辑单一，直接删除“本章小结”。

### 3.3 各章必须回答的问题

| 章节 | 必须回答的问题 |
| --- | --- |
| 绪论 | 为什么要研究这个问题，问题边界在哪里 |
| 需求分析 | 系统必须解决什么，受哪些条件约束 |
| 系统设计 | 如何把需求组织成结构与机制 |
| 系统实现 | 结构如何被落到关键链路和关键模块 |
| 系统测试 | 需求是否被验证，客观数据是什么，问题出现在哪里 |
| 总结与展望 | 已经完成了什么，还缺什么，后续应往哪里推进 |

## 4. 章节写作规约

### 4.1 绪论：消除“空对空”的宏观起手式

**写作任务**
- 交代研究对象所处的业务场景
- 指出当前做法的主要矛盾与不足
- 说明本文的目标、范围与边界

**必须杜绝的写法（AI重灾区）**
- **拒绝宏大叙事：** 严禁出现“在当今信息化时代”、“随着互联网的飞速发展”等毫无信息量的开场白。
- **直接切入业务：** 各级标题（如 1.1 背景）下的第一句话，必须直击本节的核心技术对象或业务对象。如：“成绩管理的主要矛盾不在录入本身，而在录入、审核、查询之间缺少统一数据入口。”

### 4.2 需求分析：严禁无脑机器式列举（Bullet Point 综合征）

**写作任务**
- 明确系统对象、角色边界和业务范围
- 提出功能需求与非功能需求

**必须杜绝的写法（AI重灾区）**
- **严禁词汇解释型的无脑列举：** AI 极其喜欢用 1. 2. 3. 罗列非功能需求（如“1. 可靠性：保证系统不崩溃。 2. 扩展性：系统易于扩展”）。这种脱离业务的泛泛而谈必须彻底删掉。
- **正确写法：** 非功能需求必须与业务挂钩。例如：“针对千万级并发查询瓶颈，系统可用性需求设定为 P99 延迟低于 50ms”。

### 4.3 系统设计：拒绝“技术百科全书”

**写作任务**
- 说明系统总体结构与组织逻辑
- 建立需求与结构之间的映射关系

**必须杜绝的写法（AI重灾区）**
- **严禁科普默写：** 现在的AI极喜欢默写技术说明书。**不要解释什么是 SpringBoot、什么是 Redis、什么是 AES 算法**。你的任务是写“本系统如何使用该技术解决了什么具体业务问题”。
- **错误示范：** “Redis 是一个开源的内存键值对数据库，支持多种数据结构……”
- **正确示范：** “针对高频检索带来的数据库读写锁冲突，系统引入 Redis 构建会话状态与访问令牌的二级缓存，设定 2 小时过期机制……”

### 4.4 系统实现：拒绝“产品说明书”与截图堆砌

**写作任务**
- 展示关键设计如何被落实
- 说明关键逻辑（数据流、状态流、权限流或调用关系）

**必须杜绝的写法（AI重灾区）**
- 大量毫无解释的页面截图，或逐页介绍界面的“产品操作手册”。
- 毫无意义的流水账式代码堆砌。如果贴代码，必须是体现核心设计模式、并发控制或复杂算法的关键片段，且配有深度逻辑解析。

### 4.5 系统测试：拒绝“完美主义的测试幻觉”

**写作任务**
- 以需求为对象组织测试，以客观数据为结果。

**必须杜绝的写法（AI重灾区）**
- **全篇“测试通过”：** AI 生成的测试报告往往是“天下太平”的，100% 成功，毫无瑕疵。**真实的系统不可能毫无瑕疵。**
- **强制要求：** 系统测试不仅要展示“成功路径”，必须强制报告**临界表现**或**异常触发条件**。测试数据必须包含真实测量的响应指标（并发量、内存开销、P99延迟等）。严禁使用“系统响应极快”、“运行良好”等主观形容词。

### 4.6 总结与展望

**写作任务**
- 概括已完成的工作，准确指出当前版本的主要限制。

**必须杜绝的写法**
- “由于时间有限，系统还存在许多不足。”
- “本文完成了……因此具有广泛应用前景。”
- “希望本研究能够为相关领域提供借鉴。”

## 5. 段落结构规约

### 5.1 总原则（防范AI式总分总）

1. 一段只承担一个中心任务。
2. 优先在段首句给出核心陈述或判断，再展开依据。
3. **段落末尾非必要绝不总结收束。** 把事实、逻辑和客观证据陈述完毕后，段落应当戛然而止，或直接用逻辑推导自然过渡到下一段。坚决砍掉一切诸如“总而言之”、“这意味着”、“可以看出”的段落兜底废话。工程论文不需要像初中生作文一样每段强行扣题。

### 5.2 常用段落模型

- **问题界定段：** `场景 -> 矛盾 -> 后果 -> 目标`
- **需求提炼段：** `对象 -> 行为 -> 约束`
- **设计决策段：** `原则 -> 方案 -> 映射 -> 理由`
- **实现说明段：** `目标 -> 方法 -> 关键逻辑 -> 结果`
- **测试分析段：** `对象 -> 结果（客观数据） -> 对照 -> 解释`

## 6. 句法与语意规约

### 6.1 杜绝主谓宾同义反复（车轱辘话）

AI 为了扩充文本长度，常把简单的意思扩写成极其复杂的同义反复结构。禁止在同一个句子里，用同义词互相解释。

- **AI味重灾区：** “系统通过优化数据库查询策略，极大地提升了查询的速度和效率，从而让用户能够更快速地获取所需的数据信息。”（查询策略、速度、效率、快速、获取数据，全是同义重复，毫无信息增量）。
- **矫正要求：** 直接用数据或技术实现说话。“系统通过在热门查询链路引入主从读写分离机制，使单次检索的平均耗时从 1.2s 降低至 50ms。”

### 6.2 长短句与节奏

- **长句铺关系 -> 短句落判断 -> 长句补约束。**
- 不允许连续多段长度完全趋同（高度一致的段落长度是典型的生成式文本特征）。

## 7. 高频词汇选用指南

**推荐使用（体现工程严谨性）：**
- 矛盾、瓶颈、滞后、冲突、不一致、约束、边界
- 覆盖、支撑、隔离、校验、解耦、映射、抽象、调度
- 增量更新、状态同步、异步处理、降级、幂等
- 暴露、偏离、受限于、集中在、仍依赖

**严禁使用（触发删改的标志词）：**
- 飞速发展、信息化时代
- 大大提高、显著增强、非常友好、完整完善、前景广阔
- 不仅...而且...、一方面...另一方面...（如果后续没有实质对立面）

## 8. 终稿自查清单（防AI味专项）

在提交正文前，必须逐项自查并无情删改：

1. [ ] **排雷节首起手式：** 章节的第一句话是不是“在当今时代……”？如果是，删掉，直接写技术和业务。
2. [ ] **排雷科普说明书：** 是否花了大段篇幅介绍某项流行技术（如 Docker、Vue）的定义和优点？如果是，删去，改为“本系统如何使用该技术”。
3. [ ] **排雷机器式列举：** 是否存在脱离业务场景、干瘪的“1. 2. 3.”性能指标名词解释？如果是，重写为结合业务的连贯段落。
4. [ ] **排雷段末废话总结：** 段尾是否出现了“总而言之”、“这意味着”、“由此可见该功能很重要”？如果是，直接把这句话按 Backspace 删掉。
5. [ ] **排雷完美测试：** 系统测试结果是不是全是文字上的“测试通过”？有没有明确的并发数据、响应时延或错误复现率？如果全是优美形容词，必须重写测试结果。
6. [ ] **排雷八股文小结：** 每章末尾是否有“本章首先……其次……为下一章奠定基础”？如果是，删除或改写为干货交付。
7. [ ] **排雷同义反复：** 是否存在“提高了效率和速度，加快了运行时间”这种车轱辘话？如果是，替换为客观事实或性能指标。

---

## 9. 一句话总约定

系统实现类工程论文的灵魂在于**“踩在泥土里的真实感”**。不是在证明“做了一个系统”，而是在证明“一个问题如何被界定、如何被组织、如何被落地、又如何被验证”。所有拔高的、总结的、圆滑的、四平八稳的废话，一律删掉；留下的，只能是**只有亲手踩过坑才写得出来的工程细节**。
